ISO-103 · SEMANA 5.4

Acesso à Internet via Firewall — Configuração de IP nas VMs

Retomada da semana anterior · Desktop e Server acessando internet pelo ubuntu-firewall
← Índice

🔄 Retomada — O que fizemos na semana anterior

🔥 Ubuntu Firewall — duas placas

Configuramos o ubuntu-firewall com duas interfaces de rede: enp0s3 em modo NAT (saída para internet) e enp0s8 em modo Rede Interna com IP fixo 192.168.10.1.

⚡ IP Forwarding ativado

Habilitamos o repasse de pacotes entre interfaces editando /etc/sysctl.conf e definindo net.ipv4.ip_forward=1 — sem isso o firewall não funciona como gateway.

🔒 UFW + regras NAT

Ativamos o UFW com política padrão de bloqueio e adicionamos a regra de MASQUERADE no /etc/ufw/before.rules para traduzir os IPs da rede interna ao sair.

📋 O que faltou

O firewall estava pronto para rotear, mas o ubuntu-server e o ubuntu-desktop ainda não sabiam usar o firewall como gateway. É exatamente isso que faremos hoje.

🎯 Objetivo desta aula

Configurar o ubuntu-server e o ubuntu-desktop para que ambos consigam acessar a internet roteando o tráfego pelo ubuntu-firewall. Para isso, cada VM precisa receber um IP fixo na rede interna e apontar o gateway padrão para o IP do firewall (192.168.10.1). Ao final, um ping 8.8.8.8 em qualquer uma das duas VMs deve funcionar — passando obrigatoriamente pelo firewall antes de sair para a internet.

🗺
Topologia Completa da Rede

Todo tráfego de internet do server e do desktop passa pelo firewall (enp0s8 → enp0s3 → NAT VirtualBox → Internet)

📋 Tabela de Referência Rápida
VM Interface (Rede Interna) IP Fixo Gateway DNS Usuário / Senha
🔥 ubuntu-firewall enp0s8 192.168.10.1/24 — (é o GW) — ubuntu-fw / Fatec2026
🖥 ubuntu-server enp0s3 192.168.10.10/24 192.168.10.1 8.8.8.8 ubuntu-server / Fatec2026
🖥 ubuntu-desktop enp0s3 192.168.10.20/24 192.168.10.1 8.8.8.8 ubuntu-desktop / Fatec2026
💡 Por que IP fixo nas VMs? O gateway precisa saber para qual IP encaminhar a resposta da internet. Se o IP do server ou do desktop mudar (DHCP), o roteamento pode falhar. Com IP fixo, a comunicação é sempre previsível.

Pré-Requisitos — Confirme Antes de Começar
✅ Checklist — O que deve estar pronto
  • ✓
    A VM ubuntu-firewall está ligada e com as duas interfaces configuradas (enp0s3 NAT + enp0s8 Rede Interna)
  • ✓
    O IP Forwarding está ativo no firewall — cat /proc/sys/net/ipv4/ip_forward retorna 1
  • ✓
    As regras de NAT (MASQUERADE) estão configuradas no /etc/ufw/before.rules do firewall
  • ✓
    O UFW está ativo no firewall — sudo ufw status mostra Status: active
  • ✓
    A placa de rede do ubuntu-server está em modo Rede Interna no VirtualBox
  • ✓
    A placa de rede do ubuntu-desktop está em modo Rede Interna no VirtualBox
⛔ Firewall não configurado? Volte para a Semana 5.2 e complete a configuração antes de continuar. Sem o firewall pronto, nenhuma das VMs conseguirá sair para a internet.
🔍 Verificação Rápida no Firewall (execute antes de tudo)

No terminal do 🔥 ubuntu-firewall, rode estes comandos para confirmar que está tudo ok:

# 1 — IP Forwarding ativo? fw@firewall:~$ cat /proc/sys/net/ipv4/ip_forward 1 ← deve ser 1. Se for 0, volte para a Semana 5.2 # 2 — UFW ativo? fw@firewall:~$ sudo ufw status Status: active ← deve aparecer active # 3 — Regra NAT existe? fw@firewall:~$ sudo iptables -t nat -L POSTROUTING -n MASQUERADE all -- 192.168.10.0/24 0.0.0.0/0 ← deve aparecer # 4 — Interface interna tem IP? fw@firewall:~$ ip addr show enp0s8 inet 192.168.10.1/24 ... ← deve aparecer
✅ Todos retornaram correto? Ótimo — o firewall está pronto. Agora vamos configurar o server e o desktop.
🔥
Revisão — Como o Firewall Está Configurado
📚 Por que revisar? Antes de configurar as VMs clientes, é importante entender o que o firewall faz com os pacotes que chegam da rede interna. Isso ajuda a diagnosticar problemas quando algo não funcionar.
🔄 O caminho de um pacote da internet
📥 Chegada (enp0s8)
Pacote chega com IP de origem 192.168.10.10 (server) ou 192.168.10.20 (desktop). O firewall recebe pelo enp0s8 e, por ter IP Forwarding ativo, encaminha para o enp0s3.
📤 Saída (enp0s3 + MASQUERADE)
O MASQUERADE troca o IP de origem interno pelo IP externo do enp0s3. Para a internet, parece que o próprio firewall fez o pedido. A resposta volta ao firewall, que repassa à VM correta.
1
Configurar o ubuntu-server
🖥 Contexto: Faça login na VM 🖥 ubuntu-server com usuário ubuntu-server e senha Fatec2026. Vamos configurar IP fixo e gateway via Netplan.
1
Verificar o nome da interface de rede
A interface pode se chamar enp0s3, enp0s8, eth0 ou outro nome
ubuntu-server@server:~$ ip link show
Saída esperada
1: lo: <LOOPBACK> ... ← ignorar 2: enp0s3: <BROADCAST,MULTICAST,UP,LOWER_UP> ← esta é a nossa interface
💡 Anote o nome: se aparecer enp0s3, enp0s8 ou eth0, use esse nome nos próximos passos. Nos exemplos usaremos enp0s3.
2
Verificar o arquivo Netplan existente
Confirmar o nome do arquivo antes de editar
ubuntu-server@server:~$ ls /etc/netplan/
Saída esperada
00-installer-config.yaml ← este é o arquivo a editar
⚠️ Nome diferente? Pode aparecer 01-netcfg.yaml ou outro nome. Use o nome que aparecer no seu sistema.
3
Editar o arquivo Netplan — IP fixo + gateway
IP: 192.168.10.10 · Gateway: 192.168.10.1 (firewall)
ubuntu-server@server:~$ sudo nano /etc/netplan/00-installer-config.yaml

Apague o conteúdo atual e cole o seguinte:

network: version: 2 ethernets: enp0s3: # ← troque pelo nome da SUA interface dhcp4: false # desabilita DHCP — usaremos IP fixo addresses: - 192.168.10.10/24 # IP fixo do servidor na rede interna routes: - to: default via: 192.168.10.1 # ← gateway = IP do firewall nameservers: addresses: - 8.8.8.8 # DNS Google - 1.1.1.1 # DNS Cloudflare
📐 Mapa de espaços — conte exatamente assim:

network:                          ← nível 0 — sem espaço antes
··version: 2                   ← nível 1 — 2 espaços antes
··ethernets:                  ← nível 1 — 2 espaços antes
····enp0s3:                    ← nível 2 — 4 espaços antes
······dhcp4: false             ← nível 3 — 6 espaços antes
······addresses:                ← nível 3 — 6 espaços antes
········- 192.168.10.10/24      ← nível 4 — 8 espaços + traço
······routes:                    ← nível 3 — 6 espaços antes
········- to: default            ← nível 4 — 8 espaços + traço
··········via: 192.168.10.1       ← nível 5 — 10 espaços antes
······nameservers:              ← nível 3 — 6 espaços antes
········addresses:                ← nível 4 — 8 espaços antes
··········- 8.8.8.8              ← nível 5 — 10 espaços + traço
⚠️ Use espaços, nunca a tecla Tab. Um espaço errado causa erro ao aplicar.

Para salvar no nano: Ctrl+X → Y → Enter.

4
Aplicar a configuração
Sem precisar reiniciar a VM
# Testar sintaxe antes de aplicar (opcional mas recomendado) ubuntu-server@server:~$ sudo netplan try # se der erro, reverte sozinho em 120s # Aplicar definitivamente ubuntu-server@server:~$ sudo netplan apply
⚠️ Se aparecer "invalid YAML": há um erro de indentação. Abra o arquivo novamente com sudo nano e use o mapa de espaços acima para corrigir.
5
Verificar se a configuração foi aplicada
Confirmar IP, rota e testar a saída
# Ver IP atribuído ubuntu-server@server:~$ ip addr show enp0s3 # Verificar rota padrão (gateway) ubuntu-server@server:~$ ip route show # Testar alcance até o gateway (firewall) ubuntu-server@server:~$ ping -c 2 192.168.10.1 # Testar saída para a internet ubuntu-server@server:~$ ping -c 4 8.8.8.8 # Testar resolução de nomes (DNS) ubuntu-server@server:~$ ping -c 2 google.com
Saída esperada — ip route show
default via 192.168.10.1 dev enp0s3 proto static ← gateway = firewall ✅ 192.168.10.0/24 dev enp0s3 proto kernel scope link src 192.168.10.10
✅ Sucesso! A linha default via 192.168.10.1 confirma que o firewall é o gateway padrão do servidor. Todo tráfego externo passa por ele.

Abrir terminal e verificar interface

IP: 192.168.10.20 · Gateway: 192.168.10.1

Aplicar e verificar

Saída esperada — ip route show
default via 192.168.10.1 dev enp0s3 proto static ← gateway = firewall ✅ 192.168.10.0/24 dev enp0s3 proto kernel scope link src 192.168.10.20
1
Abrir Configurações → Rede

Clique no ícone de rede na barra superior direita → Configurações de Rede.

Ou acesse pelo menu: Configurações do Sistema → Rede.

2
Editar a conexão com fio — aba IPv4

Clique na engrenagem ⚙️ ao lado da conexão com fio ativa. Vá para a aba IPv4 e selecione Manual.

Endereço: 192.168.10.20 Máscara: 255.255.255.0 (equivalente a /24) Gateway: 192.168.10.1 ← IP do firewall DNS: 8.8.8.8, 1.1.1.1

Clique em Aplicar e desligue/ligue a interface para reconectar.

⚠️ Confirme no terminal: após aplicar pela GUI, verifique com ip route show se o gateway 192.168.10.1 aparece na rota padrão.
3
Testes Finais — Validar Tudo
🔬 Execute em ambas as VMs (server e desktop). Todos os testes devem passar para confirmar que o roteamento via firewall está funcionando corretamente.
🧪 Bateria Completa de Testes
# ── TESTE 1 — IP da própria VM $ ip addr show | grep inet → server deve mostrar 192.168.10.10 | desktop: 192.168.10.20 # ── TESTE 2 — Rota padrão aponta para o firewall $ ip route show | grep default → deve mostrar: default via 192.168.10.1 # ── TESTE 3 — Alcança o firewall (gateway) $ ping -c 2 192.168.10.1 → deve responder sem perda de pacotes # ── TESTE 4 — Saída para a internet (IP) $ ping -c 4 8.8.8.8 → deve responder: 0% packet loss # ── TESTE 5 — Resolução DNS (nomes) $ ping -c 2 google.com → deve resolver o nome e responder # ── TESTE 6 — Traceroute (prova o caminho) $ traceroute -m 4 8.8.8.8 → 1º hop DEVE ser 192.168.10.1 (o firewall)
Resultado esperado — traceroute
traceroute to 8.8.8.8 (8.8.8.8), 4 hops max 1 192.168.10.1 1.1 ms ← firewall (gateway interno) ✅ 2 10.0.2.2 2.0 ms ← gateway NAT do VirtualBox 3 ... x.x ms ← roteadores da internet
✅ Prova definitiva: o 1º hop do traceroute sendo 192.168.10.1 confirma que o tráfego sai da VM, passa pelo firewall e então vai para a internet. O cenário está 100% funcional.
🔧 Troubleshooting — O que fazer se algo não funcionar
!
Ping para 192.168.10.1 falha (não alcança nem o firewall)
# Na VM com problema — ver se o IP foi atribuído $ ip addr show enp0s3 → se não aparecer inet 192.168.10.x, o netplan não foi aplicado # Reaplicar o netplan $ sudo netplan apply # No VirtualBox — confirmar que a placa está em "Rede Interna" → desligue a VM, vá em Configurações → Rede → Placa 1 → Modo: Rede Interna
!
Alcança o firewall mas não sai para a internet
# No FIREWALL — verificar IP Forwarding fw$ cat /proc/sys/net/ipv4/ip_forward → deve retornar 1. Se for 0: fw$ sudo sysctl -w net.ipv4.ip_forward=1 # No FIREWALL — verificar regra NAT fw$ sudo iptables -t nat -L POSTROUTING -n -v → deve ter MASQUERADE para 192.168.10.0/24 # No FIREWALL — verificar se o UFW está ativo fw$ sudo ufw status verbose
!
Ping para IP funciona mas DNS não resolve (google.com falha)
# Testar DNS manualmente $ nslookup google.com 8.8.8.8 # Verificar se o resolv.conf tem nameserver $ cat /etc/resolv.conf → deve ter: nameserver 8.8.8.8 # Liberar porta DNS no firewall (se necessário) fw$ sudo ufw allow out 53/udp fw$ sudo ufw allow out 53/tcp
!
Erro "invalid YAML" ao executar netplan apply
# Ver o erro detalhado $ sudo netplan --debug apply # Abrir e corrigir o arquivo $ sudo nano /etc/netplan/00-installer-config.yaml → use o Mapa de Espaços da seção anterior → verifique cada linha: 2 / 4 / 6 / 8 / 10 espaços conforme o nível → NUNCA use Tab — apenas barra de espaço
