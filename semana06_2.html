TÓPICOS DETALHADOS

1. Administração de Identidades

O Linux gerencia usuários e grupos por meio de arquivos de configuração e comandos dedicados. Entender o ciclo de vida completo é essencial para controle de acesso seguro.

# Criar usuário com home, bash e senha $ sudo useradd -m -s /bin/bash -c "Joao Silva" joao $ sudo passwd joao # Modificar usuário (adicionar a grupo, mudar shell) $ sudo usermod -aG sudo joao $ sudo usermod -s /usr/sbin/nologin joao # Excluir usuário (preservando home com -r remove tudo) $ sudo userdel joao $ sudo userdel -r joao # remove home também
📁 Arquivos sensíveis:
/etc/passwd — dados públicos (usuário:x:UID:GID:info:home:shell)
/etc/shadow — hashes de senhas e políticas (apenas root acessa)
/etc/group — definição de grupos do sistema
# Inspecionar arquivos sensíveis $ cat /etc/passwd | grep joao joao:x:1001:1001:Joao Silva:/home/joao:/bin/bash # ↑ "x" = senha no shadow ↑UID ↑GID ↑info ↑home ↑shell $ sudo cat /etc/shadow | grep joao joao:$6$salt$hash...:19720:0:90:7::: # ↑ hash SHA-512 ↑ último troca ↑ mínimo ↑ máximo ↑ aviso

2. Controle de Acesso e Permissões (rwx)

O modelo rwx aplica-se em três camadas: dono (u), grupo (g) e outros (o). Cada camada pode ter leitura (r=4), escrita (w=2) e execução (x=1).

# Verificar permissões $ ls -la /home/joao/ drwxr-xr-x 2 joao joao 4096 Apr 24 10:00 . #↑ tipo ↑ dono ↑ grupo ↑ outros # chmod notação simbólica $ chmod u+x script.sh # dono pode executar $ chmod go-w arquivo.conf # grupo e outros sem escrita $ chmod u=rwx,g=rx,o= dir/ # definição explícita # chmod notação octal $ chmod 755 diretorio_publico/ # rwxr-xr-x $ chmod 700 diretorio_privado/ # rwx------ $ chmod 644 documento.txt # rw-r--r--
# Permissões especiais $ chmod u+s /usr/bin/programa # SUID: executa como dono $ chmod g+s /empresa/ti/ # SGID: herda grupo do dir $ chmod +t /tmp # Sticky: só dono deleta $ chmod 4755 programa # SUID+755 em octal $ chmod 2770 pasta_grupo # SGID+770 $ chmod 1777 /tmp # Sticky+777
⚠️ SUID em scripts é ignorado pelo kernel Linux por segurança. Funciona apenas em binários compilados. Nunca use SUID em arquivos desnecessários — é vetor de escalada de privilégios!

3. Segurança Avançada e Delegação

O sudo permite delegar comandos específicos sem dar acesso root completo. As ACLs estendem o modelo rwx com permissões granulares por usuário ou grupo.

# Editar sudoers com segurança $ sudo visudo # Exemplos de regras no sudoers: # usuario máquina=(usuário_destino) comandos joao ALL=(ALL) NOPASSWD: /bin/systemctl restart ssh maria ALL=(ALL) /usr/bin/apt update, /usr/bin/apt upgrade %ti ALL=(ALL) NOPASSWD: /usr/bin/journalctl # Verificar privilégios do usuário atual $ sudo -l
# ACLs — Access Control Lists $ sudo apt install acl -y # Dar permissão específica a um usuário $ setfacl -m u:maria:r-- /empresa/ti/relatorio.txt # Dar permissão a um grupo $ setfacl -m g:auditoria:r-x /var/log/ # ACL padrão (novos arquivos herdam) $ setfacl -d -m g:ti:rwx /empresa/projetos/ # Consultar ACLs configuradas $ getfacl /empresa/ti/relatorio.txt user::rw- user:maria:r-- ← permissão específica! group::r-- mask::r-- other::--- # Remover ACL de um usuário $ setfacl -x u:maria /empresa/ti/relatorio.txt

4. Políticas de Conta e Senhas

O comando chage controla o ciclo de vida da senha, forçando rotação periódica — fundamental em ambientes corporativos.

# Ver políticas de senha do usuário $ sudo chage -l joao Last password change : Apr 24, 2025 Password expires : Jul 23, 2025 (90 dias) Password inactive : never Account expires : never Minimum number of days between change : 0 Maximum number of days between change : 90 Number of days of warning before expiry : 7 # Configurar política de senha (expirar em 90 dias, avisar 7 dias antes) $ sudo chage -M 90 -W 7 joao # Forçar troca de senha no próximo login $ sudo chage -d 0 joao # Expirar conta em data específica $ sudo chage -E 2025-12-31 joao # Bloquear e desbloquear conta $ sudo passwd -l joao # bloqueia $ sudo passwd -u joao # desbloqueia
# umask — máscara de criação de arquivos $ umask 0022 # Arquivos criados com 644 (666-022), diretórios com 755 (777-022) # Umask mais restritivo (ambientes seguros) $ umask 027 # arquivos=640, dirs=750 — outros sem acesso $ umask 077 # somente dono tem acesso (mais seguro)
✅ Boas práticas: Defina expiração de 90 dias para usuários comuns, 30 dias para contas privilegiadas. Sempre use chage -d 0 ao criar novas contas para forçar troca na primeira entrada.

01
👤 Ciclo de Vida de Usuários
⏱️ 20 min · useradd, usermod, userdel e análise de arquivos
+65 XP
▼
🎯 Crie usuários com parâmetros corretos, modifique e depois remova — analisando os arquivos /etc/passwd e /etc/shadow em cada etapa.
1

Crie dois usuários: aluna1 (grupo TI) e aluna2 (grupo RH) com home e bash.

$ sudo groupadd ti && sudo groupadd rh $ sudo useradd -m -s /bin/bash -G ti -c "Aluna TI" aluna1 $ sudo useradd -m -s /bin/bash -G rh -c "Aluna RH" aluna2 $ sudo passwd aluna1 && sudo passwd aluna2
2

Inspecione os arquivos sensíveis para ver os registros criados.

$ grep "aluna" /etc/passwd $ sudo grep "aluna" /etc/shadow $ grep "aluna\|ti\|rh" /etc/group
3

Modifique aluna2: adicione ao grupo TI e bloqueie a conta temporariamente.

$ sudo usermod -aG ti aluna2 $ sudo passwd -l aluna2 $ sudo grep "aluna2" /etc/shadow aluna2:!$6$hash...: ← "!" indica conta bloqueada!
4

Remova aluna1 preservando o home, depois verifique o que permanece.

$ sudo userdel aluna1 $ ls -la /home/aluna1/ drwxr-xr-x 2 1002 1002 4096 ... /home/aluna1/ # UID não resolve mais para nome — usuário removido mas home persiste! $ grep "aluna1" /etc/passwd # sem resultado = removido
✅ Ciclo de vida gerenciado! O hash com "!" no shadow indica conta bloqueada — um mecanismo de segurança comum para contas de serviço.
02
🔒 Políticas de Senha com chage
⏱️ 15 min · Rotação obrigatória, expiração e aviso
+65 XP
▼
🎯 Configure políticas de senha para garantir rotação periódica e forçar troca no primeiro login — requisito básico de auditoria e compliance.
# Verificar estado atual da conta $ sudo chage -l aluna2 Last password change : Apr 24, 2025 Password expires : never Maximum days : 99999 # Aplicar política: 90 dias máx, 1 dia mín, aviso 7 dias antes $ sudo chage -M 90 -m 1 -W 7 aluna2 # Forçar troca de senha no próximo login (data de última mudança = 0) $ sudo chage -d 0 aluna2 # Verificar resultado $ sudo chage -l aluna2 Password expires : Jul 23, 2025 Minimum number of days : 1 Number of days of warning: 7
# Testar: login como aluna2 deve pedir nova senha imediatamente $ su - aluna2 WARNING: Your password has expired. You must change your password now and login again! New password: _ # O sistema bloqueia até que a senha seja trocada!
💡 Dica de compliance: Em ambientes regulados (PCI-DSS, ISO 27001), geralmente exige-se rotação a cada 60-90 dias, histórico dos últimos 5 senhas e bloqueio após 3-5 tentativas falhas.
03
📋 ACLs para Controle Granular
⏱️ 25 min · setfacl, getfacl e permissões por usuário
+65 XP
▼
🎯 O modelo rwx padrão limita a 3 entidades (dono, grupo, outros). ACLs permitem definir acesso por usuário/grupo específico sem alterar a propriedade do arquivo.
⚠️ Verifique que o sistema de arquivos foi montado com opção acl. No Ubuntu moderno, ext4 suporta ACL por padrão.
# Criar estrutura de teste $ sudo mkdir -p /labs/documentos $ sudo bash -c 'echo "Documento confidencial" > /labs/documentos/relatorio.txt' $ sudo chown root:ti /labs/documentos/relatorio.txt $ sudo chmod 640 /labs/documentos/relatorio.txt # Agora só root e membros de "ti" têm acesso
# Conceder leitura a aluna2 (RH) via ACL $ sudo setfacl -m u:aluna2:r-- /labs/documentos/relatorio.txt # Consultar ACLs (o "+" no ls indica que há ACL ativa) $ ls -la /labs/documentos/relatorio.txt -rw-r-----+ 1 root ti 23 ... relatorio.txt # ↑ "+" = ACL ativa! $ getfacl /labs/documentos/relatorio.txt file: labs/documentos/relatorio.txt owner: root group: ti user::rw- user:aluna2:r-- ← ACL explícita! group::r-- mask::r-- other::---
# ACL padrão para diretório (novos arquivos herdam) $ sudo setfacl -d -m g:ti:rwx /labs/documentos/ $ getfacl /labs/documentos/ default:group:ti:rwx ← novos arquivos herdarão esta ACL! # Remover ACL específica $ sudo setfacl -x u:aluna2 /labs/documentos/relatorio.txt # Remover TODAS as ACLs do arquivo $ sudo setfacl -b /labs/documentos/relatorio.txt
04
🛡️ Auditoria de Permissões Excessivas
⏱️ 20 min · Encontrar e corrigir brechas de segurança
+65 XP
▼
🎯 Permissões excessivas são vetores de movimentação lateral e escalada de privilégios. Aprenda a encontrar e corrigir brechas proativamente.
# Encontrar arquivos com SUID no sistema $ find / -perm -4000 -type f 2>/dev/null /usr/bin/passwd /usr/bin/sudo /usr/bin/pkexec /usr/bin/newgrp # Analise se todos são legítimos — SUID inesperado = suspeito! # Encontrar arquivos com SGID $ find / -perm -2000 -type f 2>/dev/null # Encontrar arquivos com permissão 777 (perigo!) $ find / -perm 0777 -type f 2>/dev/null
🚨 Alerta de segurança: chmod 777 em qualquer arquivo de configuração ou script é uma brecha crítica. Um atacante com conta local pode modificar o arquivo e escalar privilégios.
# Encontrar arquivos sem dono (orphaned files — risco! $ find / -nouser -o -nogroup 2>/dev/null # Arquivos sem dono aparecem quando usuário é deletado sem -r # Devem ser revisados e atribuídos a um dono válido # Corrigir arquivo com 777 para 644 $ find /var/www -perm 0777 -exec chmod 644 {} \; # Corrigir scripts para 755 $ find /var/www -name "*.sh" -perm 0777 -exec chmod 755 {} \;
✅ Auditoria concluída! A busca por SUID/777/orphaned files deve ser parte do processo de hardening regular do sistema.

REFLEXÃO SOBRE SEGURANÇA
⚔️ Permissões como Primeira Linha de Defesa

A correta configuração de permissões não é apenas uma tarefa administrativa, mas a primeira linha de defesa contra movimentação lateral e escalada de privilégios em sistemas Linux.

Permissões excessivas são vetores comuns para exploração de vulnerabilidades. Um invasor que compromete uma conta comum com acesso irrestrito pode se mover lateralmente pelo sistema sem acionar alertas.

O Princípio do Menor Privilégio estabelece que cada usuário, processo ou serviço deve ter apenas as permissões mínimas para executar sua função — nada mais. Isso reduz drasticamente a superfície de ataque e limita o dano em caso de comprometimento.

GLOSSÁRIO RÁPIDO
useradd / usermod / userdel
Comandos para criar, modificar e remover usuários do sistema Linux
/etc/shadow
Arquivo que armazena hashes de senhas e políticas de expiração. Leitura restrita ao root
chmod (rwx)
Altera permissões de arquivos/diretórios. Notação simbólica (u+x) ou octal (755)
SUID / SGID
Bits especiais que executam um programa com os privilégios do dono ou grupo, respectivamente
Sticky Bit
Em diretórios compartilhados, impede que um usuário delete arquivos de outro. Exemplo: /tmp
visudo
Editor seguro do arquivo /etc/sudoers, valida sintaxe antes de salvar para evitar erros
setfacl / getfacl
Configura e consulta ACLs, permitindo permissões granulares por usuário ou grupo específico
chage
Gerencia políticas de expiração de senha: duração máxima, mínima, aviso e expiração da conta
umask
Máscara que define as permissões padrão subtraídas ao criar novos arquivos e diretórios
Menor Privilégio
Princípio de segurança: cada entidade deve ter apenas as permissões mínimas necessárias
