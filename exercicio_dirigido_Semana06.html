SEÇÃO A Usuários & Arquivos do Sistema Linux Q01–Q08 · 40 pts
QUESTÃO 01 Arquivos do Sistema 5 pts
Qual arquivo Linux armazena a lista de todos os usuários do sistema e pode ser lido por qualquer usuário?
  • /etc/shadow
  • /etc/passwd
  • /etc/group
  • /etc/sudoers
Resolução
✓ Resposta: B) /etc/passwd
O arquivo /etc/passwd lista todos os usuários e é legível por qualquer um. Cada linha tem o formato:
login : x : UID : GID : comentário : /home/dir : /shell
O campo x indica que a senha real está protegida no /etc/shadow. Os outros arquivos têm funções distintas: /etc/shadow guarda hashes de senha (acesso restrito), /etc/group lista grupos, e /etc/sudoers define quem pode usar sudo.
QUESTÃO 02 Senhas e Shadow 5 pts
O arquivo /etc/shadow armazena os hashes de senha. Quem tem permissão para ler esse arquivo?
  • Qualquer usuário do sistema
  • Apenas usuários do grupo "users"
  • Apenas o root (e o grupo shadow)
  • Apenas o root e o próprio dono da conta
Resolução
✓ Resposta: C) Apenas root e grupo shadow
O /etc/shadow tem permissão 640 (ou 000 em alguns sistemas): apenas root e membros do grupo shadow podem ler. Isso protege os hashes de ataques de força bruta e dicionário. Se fosse legível por todos (como o passwd antigamente), qualquer usuário poderia tentar quebrar as senhas offline.
QUESTÃO 03 UID e Root 5 pts
Como o Linux identifica que um usuário é o superusuário (root)?
  • Pelo nome "root" no arquivo /etc/passwd
  • Pelo UID igual a 0
  • Pelo GID igual a 0
  • Pelo shell /bin/bash
Resolução
✓ Resposta: B) UID igual a 0
O kernel Linux identifica o superusuário pelo número UID = 0, não pelo nome. Isso significa que renomear "root" para outro nome não remove os privilégios — o UID 0 continuará sendo superusuário. Por isso, segurança real exige controlar quem recebe UID 0, não apenas o nome da conta.
QUESTÃO 04 Criação de Usuário 5 pts
Qual parâmetro do useradd cria automaticamente a pasta /home/nomeusuario?
  • -s
  • -c
  • -G
  • -m
Resolução
✓ Resposta: D) -m
O parâmetro -m (make home) instrui o useradd a criar o diretório /home/nomeusuario e copiar os arquivos de configuração padrão de /etc/skel. Sem -m, o usuário é criado mas não tem pasta pessoal — não consegue salvar configurações nem arquivos. Os outros parâmetros: -s define o shell, -c o comentário/nome completo, -G os grupos extras.
QUESTÃO 05 Shell do Usuário 5 pts
Qual a função do parâmetro -s /bin/bash no comando useradd?
  • Define o nome completo do usuário
  • Define o shell padrão, necessário para login interativo
  • Cria a pasta pessoal do usuário
  • Adiciona o usuário a grupos extras
Resolução
✓ Resposta: B) Define o shell padrão
O parâmetro -s (shell) define qual programa é iniciado ao fazer login. Sem -s /bin/bash, o usuário pode receber um shell inválido (como /bin/sh ou nenhum), impedindo o login interativo. Para contas de serviço que não devem ter login, usa-se -s /usr/sbin/nologin ou -s /bin/false.
QUESTÃO 06 Formato /etc/passwd 5 pts
Na linha abaixo do /etc/passwd, o que representa o campo "x"?
joao:x:1001:1001:Joao Silva:/home/joao:/bin/bash
  • Que o usuário está bloqueado
  • Que a senha está armazenada em /etc/shadow
  • Que o usuário não tem senha
  • Que o campo foi removido por segurança
Resolução
✓ Resposta: B) Senha em /etc/shadow
Historicamente, o hash da senha ficava diretamente no /etc/passwd — visível para todos. Ao mover para o /etc/shadow, o campo da senha no /etc/passwd passou a exibir apenas x como marcador. Campos da linha: login : x(senha) : UID : GID : comentário : /home : /shell.
QUESTÃO 07 Grupos 5 pts
Para adicionar o usuário "ana" ao grupo "rh" sem remover seus outros grupos, qual comando usar?
  • usermod -G rh ana
  • groupadd ana rh
  • usermod -aG rh ana
  • addgroup rh ana
Resolução
✓ Resposta: C) usermod -aG rh ana
A flag -a (append) é obrigatória! O -G sozinho substitui todos os grupos secundários. Se ana estiver no grupo sudo e você executar usermod -G rh ana, ela perde o acesso sudo imediatamente. O -aG acrescenta ao grupo sem remover os existentes.
⚠ Erro clássico: esquecer o "-a" antes do "-G" e remover privilégios acidentalmente!
QUESTÃO 08 Verificar Grupos 5 pts
Qual comando exibe o UID, GID e grupos de um usuário no Linux?
  • whoami joao
  • cat /etc/group joao
  • id joao
  • ls -la joao
Resolução
✓ Resposta: C) id joao
O comando id exibe de forma compacta todas as informações de identidade do usuário. Exemplo de saída:
uid=1001(joao) gid=1001(joao) groups=1001(joao),1002(ti),27(sudo)
Mostra: UID numérico + nome, GID primário, e todos os grupos secundários. Sem argumento, mostra o usuário atual.
SEÇÃO B Permissões rwx & Comando chmod Q09–Q16 · 40 pts
QUESTÃO 09 Leitura de Permissões 5 pts
Um arquivo tem a permissão -rw-r--r--. O que o grupo pode fazer?
  • Ler, escrever e executar
  • Ler e escrever
  • Apenas ler
  • Nada — sem acesso
Resolução
✓ Resposta: C) Apenas ler
Lendo -rw-r--r-- em 3 blocos de 3:
• Dono (pos. 2-4): rw- → lê e escreve
• Grupo (pos. 5-7): r-- → somente lê
• Outros (pos. 8-10): r-- → somente lê
Em octal: 6 (dono) + 4 (grupo) + 4 (outros) = 644. Essa é a permissão padrão para a maioria dos arquivos de texto.
QUESTÃO 10 chmod Octal 5 pts
Qual é o resultado de executar chmod 700 pasta/?
  • Dono tem acesso total; grupo e outros não têm nenhum acesso
  • Todos têm acesso total à pasta
  • Dono lê e escreve; grupo e outros só leem
  • Ninguém tem acesso
Resolução
✓ Resposta: A) Dono total; grupo e outros sem acesso
Decodificando 700:
• 7 = 4(r) + 2(w) + 1(x) = rwx para o dono
• 0 = --- para o grupo
• 0 = --- para os outros
Resultado: drwx------. Ideal para diretórios pessoais e pastas com dados sigilosos.
QUESTÃO 11 chmod 600 5 pts
Um administrador quer que somente o dono leia e escreva um arquivo, e ninguém mais tenha qualquer acesso. Qual chmod é correto?
  • chmod 755 arquivo
  • chmod 644 arquivo
  • chmod 600 arquivo
  • chmod 440 arquivo
Resolução
✓ Resposta: C) chmod 600 arquivo
600 = rw-(dono) + ---(grupo) + ---(outros) = -rw-------
Aplicações comuns: chaves SSH privadas (~/.ssh/id_rsa), arquivos .env com senhas, configurações de banco de dados.
Por que não 644? O 644 deixa o grupo e outros lerem o arquivo — inaceitável para dados sigilosos.
Por que não 440? Impediria o próprio dono de escrever no arquivo.
QUESTÃO 12 chmod Simbólico 5 pts
O comando chmod u+x script.sh realiza qual ação?
  • Remove a permissão de execução de todos
  • Adiciona permissão de execução para o grupo
  • Adiciona permissão de execução para o dono do arquivo
  • Torna o arquivo somente leitura
Resolução
✓ Resposta: C) Adiciona execução para o dono
Sintaxe simbólica do chmod: [quem][operação][permissão]
• u = user (dono) | g = grupo | o = outros | a = all (todos)
• + = adiciona | - = remove | = = define exatamente
• r = leitura | w = escrita | x = execução
Assim, u+x = adiciona execução somente para o dono. Não altera grupo nem outros.
QUESTÃO 13 Permissões Perigosas 5 pts
Por que a permissão chmod 777 em um arquivo de configuração é considerada perigosa?
  • Porque impede que o root leia o arquivo
  • Porque qualquer usuário do sistema pode modificar o arquivo, facilitando ataques
  • Porque o arquivo fica invisível para outros usuários
  • Porque consome mais espaço em disco
Resolução
✓ Resposta: B) Qualquer usuário pode modificar
777 = rwxrwxrwx — todos os usuários do sistema têm permissão total. Em arquivos de configuração: qualquer um pode injetar parâmetros maliciosos. Em scripts: qualquer usuário pode substituir o conteúdo. Vetor clássico de escalada de privilégios: um atacante com conta comum modifica um script executado pelo root e obtém acesso total.
⚠ 777 quase nunca é correto. Em produção, sempre questione quem realmente precisa de escrita.
QUESTÃO 14 Permissões Comuns 5 pts
Qual permissão octal é mais adequada para diretórios públicos, onde o dono pode editar, mas outros apenas acessam?
  • 600
  • 644
  • 755
  • 777
Resolução
✓ Resposta: C) 755
755 = rwx(dono) + r-x(grupo) + r-x(outros)
O dono pode ler, escrever e entrar (rwx=7). Grupo e outros podem ler e entrar no diretório (r-x=5), mas não criar/deletar arquivos dentro. Padrão para diretórios de servidores web, páginas públicas e softwares instalados.
Por que não 644 em diretórios? O bit x em diretórios significa "entrar" — sem ele, ninguém lista o conteúdo.
QUESTÃO 15 Listagem de Permissões 5 pts
Qual comando exibe as permissões detalhadas de arquivos e diretórios, incluindo dono e grupo?
  • cat /etc/passwd
  • ls -la
  • chmod -v
  • id -l
Resolução
✓ Resposta: B) ls -la
O ls -l exibe: tipo+permissões, links, dono, grupo, tamanho, data e nome. O -a inclui arquivos ocultos (iniciados com .). Exemplo de saída:
-rw-r--r-- 1 carlos carlos 512 Mai 08 10:00 arquivo.txt
drwx------ 2 carlos carlos 4096 Mai 08 10:00 privado/
Essencial em auditorias de segurança para verificar quem tem acesso a quê.
QUESTÃO 16 Saída do ls -la 5 pts
Na saída -rwx------, o que representa o primeiro caractere "-"?
  • Que o arquivo não tem dono
  • Que o arquivo está bloqueado
  • Que é um arquivo regular (não é diretório nem link)
  • Que o arquivo tem permissão de execução negada
Resolução
✓ Resposta: C) Arquivo regular
Primeiro caractere = tipo do arquivo:
• - = arquivo regular (texto, binário, script, etc.)
• d = diretório
• l = link simbólico
• b = dispositivo de bloco (HD, pendrive)
• c = dispositivo de caractere (teclado, serial)
• p = pipe (FIFO)
• s = socket
SEÇÃO C Controle de Acesso Avançado Q17–Q20 · 20 pts
QUESTÃO 17 ACL 5 pts
Um arquivo tem permissão -rw-r-----+. O que significa o "+" no final?
  • O arquivo tem o bit SUID ativo
  • O arquivo está bloqueado para escrita
  • O arquivo possui ACLs configuradas além das permissões padrão
  • O arquivo pertence ao grupo root
Resolução
✓ Resposta: C) ACLs configuradas
O + indica que o arquivo tem ACL (Access Control List) configurada — uma camada extra de permissões além do modelo tradicional rwx. Com ACL é possível conceder acesso a um usuário específico sem alterar dono/grupo. Para visualizar: getfacl arquivo. Para configurar: setfacl -m u:fulano:r arquivo.
QUESTÃO 18 chage 5 pts
O que o comando sudo chage -d 0 joao faz?
  • Remove a conta joao após 0 dias
  • Força joao a trocar a senha no próximo login
  • Define que a senha de joao nunca expira
  • Bloqueia a conta imediatamente
Resolução
✓ Resposta: B) Força troca de senha no próximo login
O chage -d 0 define a data da última mudança de senha como 0 (epoch Unix = 01/01/1970). O sistema interpreta que a senha está vencida desde sempre, forçando o usuário a criar uma nova senha imediatamente no próximo login. Muito útil ao criar contas: o admin define uma senha provisória e garante que o usuário a troque.
QUESTÃO 19 Auditoria 5 pts
Por que o comando find / -perm 0777 -type f é importante em auditorias de segurança?
  • Localiza arquivos de configuração para backup
  • Lista scripts que precisam de permissão de execução
  • Identifica arquivos com permissão total para todos — possíveis vetores de ataque
  • Encontra arquivos pertencentes ao root
Resolução
✓ Resposta: C) Vetores de ataque com permissão total
Arquivos com permissão 777 são alvos prioritários em auditorias. Qualquer usuário do sistema pode escrever neles. Se um script com 777 é executado periodicamente por root (via cron, por exemplo), um atacante com conta comum pode injetar código malicioso e obter escalada de privilégios.
$ find /tmp -perm 0777 -type f 2>/dev/null
/tmp/arquivo_inseguro.sh
/tmp/config_exposta.txt
QUESTÃO 20 Princípio do Menor Privilégio 5 pts
Qual afirmação descreve corretamente o Princípio do Menor Privilégio aplicado a permissões Linux?
  • Todo usuário deve ter permissão de leitura em todos os arquivos do sistema
  • Cada usuário ou processo deve ter apenas as permissões mínimas para executar sua função
  • O root deve ser usado para todas as operações a fim de evitar erros de permissão
  • Grupos devem ter sempre permissão rwx para facilitar o trabalho colaborativo
Resolução
✓ Resposta: B) Permissões mínimas para executar a função
Princípio do Menor Privilégio (PoLP): conceder apenas o acesso estritamente necessário, nada mais.
• A ≠ — acesso irrestrito aumenta a superfície de ataque
• C ≠ — usar root para tudo é perigoso; um erro derruba o sistema
• D ≠ — rwx indiscriminado para grupos viola o princípio
Por que importa: se uma conta é comprometida, o atacante só tem os privilégios daquela conta — o dano é limitado. Com root para tudo, a invasão seria total.

SEÇÃO A Criação e Gerenciamento de Usuários Tarefa 01 · 30 pts
TAREFA 01 Criando Usuários e Verificando o Sistema 30 pts
Você foi contratado como técnico de TI de uma pequena empresa. Antes de configurar o servidor, precisa criar contas para dois novos funcionários e verificar como o Linux organiza essas informações.
Execute os passos abaixo no terminal e registre os outputs no documento de entrega:
  1. Execute cat /etc/passwd | head -5. Identifique os campos: login, "x", UID, GID, comentário, home e shell.
  2. Crie o usuário carlos com pasta pessoal, shell bash e nome completo "Carlos Souza":
    sudo useradd -m -s /bin/bash -c "Carlos Souza" carlos
    Em seguida: sudo passwd carlos
  3. Crie o usuário patricia da mesma forma e defina uma senha para ela.
  4. Execute grep carlos /etc/passwd e grep patricia /etc/passwd. Identifique o UID de cada um.
  5. Execute ls -la /home/. Confirme que as pastas de carlos e patricia foram criadas automaticamente.
☑ Checklist de entrega
  • Descrição do formato do /etc/passwd com os 7 campos identificados
  • Usuários carlos e patricia criados corretamente com -m e -s /bin/bash
  • Saída do grep mostrando as linhas dos dois usuários
  • UIDs de carlos e patricia anotados (esperado: ≥ 1000)
  • Print/saída do ls -la /home/ evidenciando as pastas criadas
Resolução — Tarefa 01
Passo 1 — Formato do /etc/passwd:
$ cat /etc/passwd | head -5
root:x:0:0:root:/root:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
# Campos: login : x(senha) : UID : GID : comentário : /home : /shell
Passo 2 e 3 — Criar carlos e patricia:
$ sudo useradd -m -s /bin/bash -c "Carlos Souza" carlos
# Silêncio = sucesso no Linux
$ sudo passwd carlos
Nova senha: ****
Redigite a nova senha: ****
passwd: senha atualizada com sucesso

$ sudo useradd -m -s /bin/bash -c "Patricia Lima" patricia
$ sudo passwd patricia
Passo 4 — Verificar no /etc/passwd:
$ grep carlos /etc/passwd
carlos:x:1001:1001:Carlos Souza:/home/carlos:/bin/bash
$ grep patricia /etc/passwd
patricia:x:1002:1002:Patricia Lima:/home/patricia:/bin/bash
Passo 5 — Confirmar as pastas pessoais:
$ ls -la /home/
drwxr-xr-x 4 root root 4096 Mai 08 10:00 .
drwx------ 15 carlos carlos 4096 Mai 08 10:01 carlos
drwx------ 15 patricia patricia 4096 Mai 08 10:02 patricia
💡 O -m cria o diretório home automaticamente. O dono da pasta é o próprio usuário (drwx------ = 700). Isso garante privacidade entre usuários do mesmo sistema.
SEÇÃO B Grupos e Permissões de Acesso Tarefa 02 · 30 pts
TAREFA 02 Criando Grupos e Controlando o Acesso a Arquivos 30 pts
A empresa tem dois departamentos: TI e RH. Crie grupos para cada setor, associe os usuários e garanta que arquivos sigilosos só sejam acessados pelo grupo correto.
  1. Crie os grupos ti e rh:
    sudo groupadd ti && sudo groupadd rh
  2. Adicione carlos ao grupo ti e patricia ao grupo rh sem remover grupos originais:
    sudo usermod -aG ti carlos
    sudo usermod -aG rh patricia
  3. Execute id carlos e id patricia. Confirme os grupos e cole a saída.
  4. Crie um arquivo sigiloso com permissão somente para o dono:
    touch /tmp/folha_pagamento.txt
    chmod 600 /tmp/folha_pagamento.txt
  5. Execute ls -la /tmp/folha_pagamento.txt. Explique por escrito o que cada parte da permissão significa.
☑ Checklist de entrega
  • Grupos ti e rh criados com groupadd
  • carlos no grupo ti e patricia no grupo rh (via usermod -aG)
  • Saída do id de cada usuário confirmando os grupos
  • Arquivo /tmp/folha_pagamento.txt com chmod 600 aplicado
  • Saída do ls -la colada e permissão explicada por escrito
Resolução — Tarefa 02
Passo 1 — Criar grupos:
$ sudo groupadd ti && sudo groupadd rh
# Silêncio = grupos criados com sucesso
$ grep -E "^ti:|^rh:" /etc/group
ti:x:1003:
rh:x:1004:
Passo 2 — Associar usuários aos grupos (CUIDADO com -aG):
$ sudo usermod -aG ti carlos
$ sudo usermod -aG rh patricia
# -a = append (acrescenta). Sem "a", substitui todos os grupos!
Passo 3 — Verificar com id:
$ id carlos
uid=1001(carlos) gid=1001(carlos) groups=1001(carlos),1003(ti)
$ id patricia
uid=1002(patricia) gid=1002(patricia) groups=1002(patricia),1004(rh)
Passos 4 e 5 — Arquivo sigiloso com chmod 600:
$ touch /tmp/folha_pagamento.txt
$ chmod 600 /tmp/folha_pagamento.txt
$ ls -la /tmp/folha_pagamento.txt
-rw------- 1 aluno aluno 0 Mai 08 10:05 /tmp/folha_pagamento.txt
Explicação da permissão -rw-------:
• - → arquivo regular
• rw- → dono: lê e escreve (sem execução)
• --- → grupo: sem acesso nenhum
• --- → outros: sem acesso nenhum
Em octal: 6+0+0 = 600. Ideal para folhas de pagamento, chaves SSH, e arquivos com senhas.
SEÇÃO C Estrutura de Diretórios com Permissões Profissionais Tarefa 03 · 25 pts
TAREFA 03 Criando a Estrutura de Arquivos da Empresa com Permissões Corretas 25 pts

Passo 5 — Justificativa:
O diretório rh/ recebe 700 (drwx------) porque contém dados sigilosos: salários, contratos, dados pessoais. Com permissão 755, qualquer usuário do sistema poderia entrar no diretório e listar os arquivos — mesmo sem poder lê-los diretamente, já saberia quais arquivos existem, o que viola a confidencialidade. A permissão 700 garante que apenas o dono (administrador) possa acessar o diretório.

TAREFA 04 Detectando e Corrigindo Permissões Inseguras 15 pts
Você recebeu um servidor com suspeita de configurações incorretas de permissão. Precisa identificar arquivos com permissão total (777) e corrigi-los.
  1. Crie dois arquivos com permissão insegura para simular a situação:
    touch /tmp/arquivo_inseguro.sh && chmod 777 /tmp/arquivo_inseguro.sh
    touch /tmp/config_exposta.txt && chmod 777 /tmp/config_exposta.txt
  2. Execute a auditoria para encontrar arquivos com permissão 777 em /tmp:
    find /tmp -perm 0777 -type f 2>/dev/null
  3. Corrija as permissões: chmod 644 em config_exposta.txt e chmod 755 em arquivo_inseguro.sh.
  4. Rode o find novamente. Cole a saída e confirme que nenhum arquivo com 777 aparece.
  5. Explique por escrito: qual é o risco de um script .sh com permissão 777 em servidor com múltiplos usuários?

Resolução — Tarefa 04
Passo 1 — Simular permissões inseguras:
$ touch /tmp/arquivo_inseguro.sh && chmod 777 /tmp/arquivo_inseguro.sh
$ touch /tmp/config_exposta.txt && chmod 777 /tmp/config_exposta.txt
$ ls -la /tmp/arquivo_inseguro.sh /tmp/config_exposta.txt
-rwxrwxrwx 1 aluno aluno 0 Mai 08 /tmp/arquivo_inseguro.sh
-rwxrwxrwx 1 aluno aluno 0 Mai 08 /tmp/config_exposta.txt
Passo 2 — Auditoria (antes da correção):
$ find /tmp -perm 0777 -type f 2>/dev/null
/tmp/arquivo_inseguro.sh
/tmp/config_exposta.txt
Passo 3 — Corrigir permissões:
$ chmod 644 /tmp/config_exposta.txt
# 644 = rw-r--r-- : dono lê/escreve, grupo e outros só leem
$ chmod 755 /tmp/arquivo_inseguro.sh
# 755 = rwxr-xr-x : dono tem tudo, outros só executam/leem
Passo 4 — Auditoria (após a correção — deve retornar vazia):
$ find /tmp -perm 0777 -type f 2>/dev/null
# Saída vazia = nenhum arquivo inseguro encontrado ✓
Passo 5 — Justificativa sobre o risco de 777 em scripts:
Um script .sh com permissão 777 (-rwxrwxrwx) permite que qualquer usuário do sistema modifique seu conteúdo. Em um servidor com múltiplos usuários:
1. Um atacante com conta comum pode injetar código malicioso no script
2. Se o script for executado periodicamente por root (via cron) ou por outro usuário privilegiado, o código injetado roda com esses privilégios
3. O atacante obtém escalada de privilégios — acesso root sem precisar da senha
Esse é um dos vetores mais comuns de comprometimento em servidores Linux mal configurados.
⚠ Regra de ouro: scripts executáveis devem ter no máximo 755 (dono escreve, outros só executam).
