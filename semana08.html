📖 GLOSSÁRIO TÉCNICO
Zona
Porção do espaço de nomes DNS administrada por um servidor autoritativo
SOA
Start of Authority — registro inicial que define parâmetros da zona
TTL
Time To Live — tempo em segundos que um registro pode ficar em cache
Forwarder
Servidor DNS para onde encaminhamos queries que não conseguimos responder
Recursão
Capacidade do servidor de buscar a resposta completa por conta do cliente
DNSSEC
DNS Security Extensions — adiciona assinatura digital aos registros DNS

01
Instalação e Configuração Base do BIND9
⏱️ 20 min · Dificuldade: ⭐⭐
+60 XP
▼
🎯 Instalar BIND9, verificar serviço e configurar opções básicas
📦 INSTALAÇÃO
$ sudo apt update && sudo apt install -y bind9 bind9utils bind9-doc
# bind9 = servidor DNS  bind9utils = ferramentas  bind9-doc = documentação

$ sudo systemctl enable --now named
# Habilita e inicia o serviço (named = nome do daemon BIND9)

$ sudo systemctl status named
● named.service - BIND Domain Name Server
   Loaded: loaded (/lib/systemd/system/named.service; enabled)
   Active: active (running) since ...

$ sudo named-checkconf
# Verifica sintaxe da configuração — silêncio = OK
⚙️ /etc/bind/named.conf.options — CONFIGURAR FORWARDERS
# Editar o arquivo de opções globais:
$ sudo nano /etc/bind/named.conf.options

options {
    directory "/var/cache/bind";

    forwarders {
        8.8.8.8;    // Google DNS primário
        8.8.4.4;    // Google DNS secundário
        1.1.1.1;    // Cloudflare
    };

    dnssec-validation auto;
    listen-on { any; };
    allow-query { any; };
    recursion yes;
};

$ sudo named-checkconf && sudo systemctl restart named
# Verifica e reinicia após cada alteração
⚠️ Firewall: Libere a porta 53 UDP e TCP: sudo ufw allow 53
✅ Teste rápido: dig @localhost google.com — deve retornar resposta com status NOERROR
02
Criando Zona Direta (example.com)
⏱️ 30 min · Dificuldade: ⭐⭐⭐
+70 XP
▼
🎯 Criar zona autoritativa para example.com com múltiplos registros
📝 PASSO 1 — Declarar zona em named.conf.local
$ sudo nano /etc/bind/named.conf.local

zone "example.com" {
    type master;
    file "/etc/bind/db.example.com";
    allow-transfer { none; };    // Segurança: negar transferência de zona
};
📄 PASSO 2 — Criar arquivo de zona /etc/bind/db.example.com
$ sudo cp /etc/bind/db.local /etc/bind/db.example.com
$ sudo nano /etc/bind/db.example.com

; Arquivo de zona para example.com
; Última modificação: 2024-01-15
$TTL    604800       ; TTL padrão: 7 dias

@    IN    SOA    ns1.example.com. admin.example.com. (
                  2024011501    ; Serial (ANO+MÊS+DIA+REVISÃO)
                  3600          ; Refresh (1h)
                  900           ; Retry (15min)
                  604800        ; Expire (7 dias)
                  86400 )       ; Negative TTL (1 dia)

; Nameservers
@    IN    NS    ns1.example.com.
@    IN    NS    ns2.example.com.

; Registros A (Host → IPv4)
@              IN    A       192.168.1.10
ns1            IN    A       192.168.1.10
ns2            IN    A       192.168.1.11
www            IN    A       192.168.1.10
mail           IN    A       192.168.1.20
ftp            IN    A       192.168.1.30

; Registros CNAME (Alias)
webmail        IN    CNAME   mail.example.com.
smtp           IN    CNAME   mail.example.com.

; Registro MX (Servidor de E-mail)
@    IN    MX    10    mail.example.com.

; Registro TXT (SPF e outros)
@    IN    TXT   "v=spf1 mx a ~all"
✅ PASSO 3 — Verificar e reiniciar
$ sudo named-checkzone example.com /etc/bind/db.example.com
zone example.com/IN: loaded serial 2024011501
OK

$ sudo systemctl restart named
$ dig @localhost www.example.com
...
;; ANSWER SECTION:
www.example.com.    604800    IN    A    192.168.1.10

$ dig @localhost mail.example.com MX
# Deve mostrar o registro MX com prioridade 10
⚠️ Serial: Sempre incremente o serial (2024011501 → 2024011502) ao modificar a zona, caso contrário servidores secundários não sincronizam.
03
Criando Zona Reversa (PTR)
⏱️ 25 min · Dificuldade: ⭐⭐⭐
+65 XP
▼
🎯 Configurar resolução reversa: IP → hostname via registros PTR
📝 PASSO 1 — Declarar zona reversa em named.conf.local
# Zona reversa para a rede 192.168.1.0/24
# Nome = rede invertida + .in-addr.arpa
zone "1.168.192.in-addr.arpa" {
    type master;
    file "/etc/bind/db.192.168.1";
};
📄 PASSO 2 — Criar arquivo de zona reversa
$ sudo nano /etc/bind/db.192.168.1

$TTL    604800
@    IN    SOA    ns1.example.com. admin.example.com. (
                  2024011501 3600 900 604800 86400 )

@    IN    NS    ns1.example.com.

; Registros PTR (IP final → hostname completo com ponto final!)
10   IN    PTR   ns1.example.com.
11   IN    PTR   ns2.example.com.
20   IN    PTR   mail.example.com.
30   IN    PTR   ftp.example.com.
🔍 PASSO 3 — Testar resolução reversa
$ sudo named-checkzone 1.168.192.in-addr.arpa /etc/bind/db.192.168.1
zone 1.168.192.in-addr.arpa/IN: loaded serial 2024011501
OK

$ sudo systemctl restart named

# Resolução reversa: -x = reverse lookup
$ dig @localhost -x 192.168.1.10
;; ANSWER SECTION:
10.1.168.192.in-addr.arpa.  604800  IN  PTR  ns1.example.com.

$ nslookup 192.168.1.20 localhost
Server: localhost
Address: 127.0.0.1#53
20.1.168.192.in-addr.arpa  name = mail.example.com.
⚠️ Os hostnames nos registros PTR devem terminar com ponto final (ex: mail.example.com.) — sem o ponto, o BIND adiciona o nome da zona, criando FQDNs incorretos.
04
Testes, dig Avançado e Troubleshooting
⏱️ 20 min · Dificuldade: ⭐⭐
+55 XP
▼
🎯 Dominar as ferramentas dig e nslookup para validar e diagnosticar DNS
🔍 DIG — CONSULTAS AVANÇADAS
# Consulta básica (registro A)
$ dig @192.168.1.10 www.example.com

# Consulta específica por tipo
$ dig @localhost example.com MX
$ dig @localhost example.com NS
$ dig @localhost example.com TXT
$ dig @localhost example.com SOA

# Output curto (apenas a resposta)
$ dig @localhost www.example.com +short
192.168.1.10

# AXFR — transferência de zona (debug)
$ dig @localhost example.com AXFR

# Verificar propagação de TTL
$ dig @8.8.8.8 example.com +ttl
🚨 TROUBLESHOOTING COMUM
# Verificar logs do BIND9
$ sudo journalctl -u named -n 50 --no-pager
$ sudo tail -f /var/log/syslog | grep named

# Sintaxe de configuração
$ sudo named-checkconf -z
# -z = também verifica todos os arquivos de zona

# Teste de conectividade na porta 53
$ ss -tulnp | grep :53
udp   UNCONN  0  0  0.0.0.0:53  0.0.0.0:*  users:(("named",...))

# Flush do cache DNS
$ sudo rndc flush
$ sudo rndc reload
# rndc = remote name daemon control

# Problemas comuns:
SERVFAIL   → erro na zona ou forwarder inacessível
NXDOMAIN   → nome não existe na zona
REFUSED    → ACL bloqueando a query
connection refused → named não está rodando na porta 53
✅ Checklist final DNS: named status = active · named-checkconf sem erros · dig @localhost retorna NOERROR · resolução direta e reversa funcionando · porta 53 UDP+TCP aberta
