01
👤 Criar Usuários e Grupos Organizacionais
⏱️ 25 min · Estrutura completa para um departamento fictício
+60 XP
▼
🎯 Você é o administrador de uma empresa fictícia. Crie a estrutura de usuários e grupos para os departamentos de TI, RH e Financeiro.
# PASSO 1 — Criar os grupos departamentais admin@servidor:~$ sudo groupadd ti admin@servidor:~$ sudo groupadd rh admin@servidor:~$ sudo groupadd financeiro # Verificar grupos criados admin@servidor:~$ tail -3 /etc/group ti:x:1001: rh:x:1002: financeiro:x:1003:
# PASSO 2 — Criar 5 usuários com home directory (-m) e shell bash (-s) admin@servidor:~$ sudo useradd -m -s /bin/bash -g ti joao admin@servidor:~$ sudo useradd -m -s /bin/bash -g rh maria admin@servidor:~$ sudo useradd -m -s /bin/bash -g financeiro pedro admin@servidor:~$ sudo useradd -m -s /bin/bash -g ti ana admin@servidor:~$ sudo useradd -m -s /bin/bash -g rh carlos # Definir senha para joao (repita para os demais) admin@servidor:~$ sudo passwd joao New password: **** passwd: password updated successfully
⚠️ Sem -m o diretório home NÃO é criado. Sem -s /bin/bash o usuário pode não conseguir login interativo.
# PASSO 3 — Adicionar joao ao grupo sudo (acesso administrativo) # ATENÇÃO: use -aG (append) — sem o -a você SUBSTITUI todos os grupos! admin@servidor:~$ sudo usermod -aG sudo joao admin@servidor:~$ id joao uid=1001(joao) gid=1001(ti) groups=1001(ti),27(sudo) # PASSO 4 — Verificar /etc/passwd (sem senhas reais!) e /etc/shadow admin@servidor:~$ grep joao /etc/passwd joao:x:1001:1001::/home/joao:/bin/bash # O "x" no campo da senha significa: a senha real está no /etc/shadow admin@servidor:~$ sudo grep joao /etc/shadow joao:$6$xyz...:19500:0:99999:7::: # $6$ = hash SHA-512 — NUNCA armazena a senha em texto puro!
✅ Estrutura criada! Usuários no /etc/passwd, senhas hashadas no /etc/shadow.
02
🔐 Configurar Permissões Unix (chmod/chown)
⏱️ 30 min · Permissões básicas e bits especiais SUID/SGID/Sticky
+65 XP
▼
🎯 Configure diretórios departamentais com permissões corretas, incluindo SGID para herança de grupo e Sticky Bit para proteção de arquivos.
# PASSO 1 — Criar estrutura de diretórios da empresa admin@servidor:~$ sudo mkdir -p /empresa/{ti,rh,financeiro,publico} # PASSO 2 — Atribuir grupos corretos a cada diretório admin@servidor:~$ sudo chown root:ti /empresa/ti admin@servidor:~$ sudo chown root:rh /empresa/rh admin@servidor:~$ sudo chown root:financeiro /empresa/financeiro # PASSO 3 — Permissões 770: dono=rwx, grupo=rwx, outros=--- (privado do depto) admin@servidor:~$ sudo chmod 770 /empresa/ti /empresa/rh /empresa/financeiro # 7 = 4(r)+2(w)+1(x) = rwx | 0 = --- (sem acesso para outros) # Pasta pública: todos podem acessar (777) admin@servidor:~$ sudo chmod 777 /empresa/publico
# PASSO 4 — SGID no diretório TI: novos arquivos herdam o grupo admin@servidor:~$ sudo chmod g+s /empresa/ti admin@servidor:~$ ls -la /empresa/ drwxrws--- 2 root ti 4096 ... ti ↑ "s" = SGID ativo — arquivos novos terão grupo "ti" automaticamente! # PASSO 5 — Sticky Bit na pasta pública # Cada usuário só deleta SEUS próprios arquivos admin@servidor:~$ sudo chmod +t /empresa/publico admin@servidor:~$ ls -la /empresa/ drwxrwxrwt 2 root root 4096 ... publico ↑ "t" = Sticky Bit — protege arquivos de outros usuários! # Confirmar com notação octal completa admin@servidor:~$ stat -c "%a %n" /empresa/* 2770 /empresa/ti ← 2=SGID + 770 1777 /empresa/publico ← 1=Sticky + 777
✅ Permissões configuradas! SGID garante herança de grupo. Sticky Bit protege a pasta compartilhada.
03
🛡️ Configurar Sudo com Delegação Segura
⏱️ 20 min · Princípio do menor privilégio na prática
+65 XP
▼
🎯 Configure o sudo para que cada usuário execute APENAS os comandos que precisa — sem acesso total de root.
⚠️ NUNCA edite /etc/sudoers diretamente com nano/vim! Use sempre sudo visudo — ele valida a sintaxe antes de salvar, evitando travar o sistema.
# PASSO 1 — Abrir o editor seguro do sudoers admin@servidor:~$ sudo visudo # Isso abre /etc/sudoers no editor padrão (geralmente nano no Ubuntu) ## PASSO 2 — Adicionar ao FINAL do arquivo (antes da linha #includedir): ## joao (TI) pode reiniciar serviços de rede sem precisar digitar senha joao ALL=(ALL) NOPASSWD: /bin/systemctl restart networking, /bin/systemctl restart ssh ## maria (RH) pode consultar logs do sistema maria ALL=(ALL) NOPASSWD: /bin/journalctl, /bin/cat /var/log/syslog ## Grupo TI pode gerenciar pacotes (com %) %ti ALL=(ALL) NOPASSWD: /usr/bin/apt update, /usr/bin/apt upgrade ## Salvar: Ctrl+X → Y → Enter
# PASSO 3 — Testar como joao admin@servidor:~$ su - joao joao@servidor:~$ sudo systemctl restart ssh [Sem pedir senha — acesso liberado conforme regra!] # joao NÃO pode criar usuários (não está na regra) joao@servidor:~$ sudo useradd hacker Sorry, user joao is not allowed to execute '/usr/sbin/useradd hacker' as root ✓ Bloqueado corretamente! # PASSO 4 — Auditoria: tudo que é feito via sudo é registrado em log admin@servidor:~$ sudo journalctl | grep sudo | tail -5 servidor sudo: joao: TTY=pts/0 COMMAND=/bin/systemctl restart ssh # Rastreabilidade total — você sabe quem fez o quê e quando!
✅ Sudo configurado com menor privilégio! Cada ação fica registrada em log de auditoria.
04
📋 Implementar ACLs para Controle Granular
⏱️ 25 min · Permissões além do modelo rwx básico
+65 XP
▼
🎯 ACLs permitem dar permissões específicas a usuários individuais sem alterar dono ou grupo do arquivo. Indispensável em ambientes corporativos.
# PASSO 1 — Instalar ferramentas ACL (se necessário) admin@servidor:~$ sudo apt install acl -y # PASSO 2 — Criar arquivo de teste na pasta TI admin@servidor:~$ sudo bash -c 'echo "Relatório confidencial TI" > /empresa/ti/relatorio.txt' admin@servidor:~$ ls -la /empresa/ti/relatorio.txt -rw-r----- 1 root ti 26 ... relatorio.txt # pedro (financeiro) ainda NÃO tem acesso!
# PASSO 3 — Dar acesso de leitura APENAS para pedro via ACL admin@servidor:~$ sudo setfacl -m u:pedro:r-- /empresa/ti/relatorio.txt # Verificar a ACL configurada admin@servidor:~$ getfacl /empresa/ti/relatorio.txt file: empresa/ti/relatorio.txt owner: root group: ti user::rw- user:pedro:r-- ← pedro tem leitura explicitamente! group::r-- mask::r-- other::--- # PASSO 4 — Testar como pedro admin@servidor:~$ su - pedro -c "cat /empresa/ti/relatorio.txt" Relatório confidencial TI ← pedro consegue ler! admin@servidor:~$ su - pedro -c "echo teste >> /empresa/ti/relatorio.txt" bash: /empresa/ti/relatorio.txt: Permission denied ← mas não pode escrever! # PASSO 5 — ACL padrão: novos arquivos na pasta TI herdam a ACL admin@servidor:~$ sudo setfacl -d -m u:pedro:r-x /empresa/ti # -d = default = herança automática para novos arquivos # Arquivos com ACL mostram "+" na listagem admin@servidor:~$ ls -la /empresa/ti/ -rw-r-----+ 1 root ti 26 ... relatorio.txt ← "+" = tem ACL!
✅ ACL implementada! Controle granular além do chmod básico — perfeito para ambientes corporativos.
