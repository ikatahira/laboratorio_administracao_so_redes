💡 Solução dos Cenários de Troubleshooting

🔴 Cenário 1: Usuário não consegue acessar compartilhamento de rede

Sintoma: Ao tentar acessar \\servidor\dados, recebe erro "Caminho de rede não encontrado"

Metodologia de Resolução:

  1. Identificação: Problema de conectividade de rede ou compartilhamento
  2. Coleta de Informações:
    # No cliente ping servidor nslookup servidor net view \\servidor ipconfig /all # No servidor (verificar se Samba/SMB está rodando) # Linux: systemctl status smbd systemctl status nmbd # Windows: Get-Service LanmanServer Get-SmbShare
  3. Hipóteses:
    • Servidor não está acessível (rede)
    • Serviço de compartilhamento parado
    • Firewall bloqueando portas 139/445
    • Permissões incorretas
  4. Testes:
    # Testar conectividade ping servidor telnet servidor 445 # Verificar firewall no servidor # Linux: sudo ufw status sudo iptables -L | grep -E '139|445' # Windows: Test-NetConnection -ComputerName servidor -Port 445 Get-NetFirewallRule | Where-Object {$_.DisplayName -like "*File*"}
  5. Solução Típica:
    # Se serviço parado (Linux): sudo systemctl restart smbd nmbd # Se firewall bloqueando (Linux): sudo ufw allow samba # Windows: Restart-Service LanmanServer New-NetFirewallRule -DisplayName "File Sharing" -Direction Inbound -Protocol TCP -LocalPort 445 -Action Allow

🔴 Cenário 2: DNS não está resolvendo nomes

Sintoma: Ping google.com retorna "could not find host", mas ping 8.8.8.8 funciona

Diagnóstico e Solução:

# 1. Verificar configuração DNS do cliente ipconfig /all # Windows cat /etc/resolv.conf # Linux # 2. Testar DNS nslookup google.com nslookup google.com 8.8.8.8 # Testar com DNS público # 3. Se DNS local não funciona, verificar servidor DNS # No servidor DNS Linux: systemctl status named # ou bind9 journalctl -u named -n 50 # Verificar arquivo de zona named-checkzone example.com /etc/bind/db.example.com # Windows: Get-Service DNS Get-DnsServerZone Test-DnsServer -IPAddress 192.168.1.10 # 4. Solução comum: # Linux - reiniciar BIND sudo systemctl restart named # Windows - limpar cache DNS ipconfig /flushdns # Cliente Clear-DnsServerCache # Servidor

🔴 Cenário 3: Servidor lento - alta utilização de CPU

Sintoma: Aplicações lentas, servidor respondendo devagar

Diagnóstico:

# Linux top htop ps aux --sort=-%cpu | head -10 sar -u 1 10 # Monitorar CPU # Identificar processo problemático ps aux | grep lsof -p # Arquivos abertos pelo processo # Windows PowerShell Get-Process | Sort-Object CPU -Descending | Select-Object -First 10 Get-Counter '\Processor(_Total)\% Processor Time' -SampleInterval 1 -MaxSamples 10 # Verificar serviços Get-Service | Where-Object {$_.Status -eq "Running"} | Sort-Object # 5. Solução: # Se processo malicioso ou travado: kill -9 # Linux Stop-Process -Id -Force # Windows # Se serviço legítimo consumindo muito: systemctl restart # Linux Restart-Service # Windows

🔴 Cenário 4: Usuários não conseguem fazer login no domínio

Sintoma: "The trust relationship between this workstation and the domain failed"

Solução:

# 1. Verificar conectividade com DC Test-ComputerSecureChannel -Verbose # 2. Se falhar, reparar: Test-ComputerSecureChannel -Repair -Credential (Get-Credential) # 3. Se não funcionar, remover e readicionar ao domínio: # Remover do domínio Remove-Computer -UnjoinDomainCredential (Get-Credential) -WorkgroupName WORKGROUP -Restart # Após reiniciar, adicionar novamente Add-Computer -DomainName empresa.local -Credential (Get-Credential) -Restart # 4. No servidor DC, verificar: Get-ADComputer -Identity COMPUTADOR-NOME # Se objeto órfão, remover: Remove-ADComputer -Identity COMPUTADOR-NOME

🔴 Cenário 5: DHCP não está distribuindo IPs

Sintoma: Clientes recebendo IP 169.254.x.x (APIPA)

Diagnóstico e Solução:

# 1. Verificar serviço DHCP # Linux: systemctl status isc-dhcp-server journalctl -u isc-dhcp-server -n 50 # Windows: Get-Service DHCPServer Get-DhcpServerv4Scope Get-DhcpServerv4Lease -ScopeId 192.168.1.0 # 2. Verificar configuração # Linux: sudo nano /etc/dhcp/dhcpd.conf sudo dhcpd -t # Testar configuração # Windows: Get-DhcpServerv4Scope | Format-List * Get-DhcpServerv4ScopeStatistics # 3. Verificar autorização (Windows): Get-DhcpServerInDC # Se não autorizado: Add-DhcpServerInDC -DnsName srv-dhcp.empresa.local -IPAddress 192.168.1.10 # 4. Verificar se escopo tem IPs disponíveis: # Se pool esgotado, aumentar range: Set-DhcpServerv4Scope -ScopeId 192.168.1.0 -EndRange 192.168.1.250 # 5. Reiniciar serviço: sudo systemctl restart isc-dhcp-server # Linux Restart-Service DHCPServer # Windows # 6. No cliente, renovar IP: sudo dhclient -r && sudo dhclient # Linux ipconfig /release && ipconfig /renew # Windows
💡 METODOLOGIA PROFISSIONAL DE TROUBLESHOOTING:
  1. Identificar o problema: Definir sintomas claramente
  2. Coletar informações: Logs, comandos de diagnóstico, histórico
  3. Formular hipóteses: Listar possíveis causas (da mais provável à menos)
  4. Testar hipóteses: Um teste por vez, documentando resultados
  5. Implementar solução: Aplicar correção validada
  6. Verificar: Confirmar que problema foi resolvido
  7. Documentar: Criar KB (Knowledge Base) para futuros incidentes
  8. Revisar: Análise pós-incidente, prevenção

Projeto Final - Implementação

📋 Objetivo: Montar ambiente completo

⏱️ Duração: 60 minutos

📄 Descrição: Infraestrutura com Linux e Windows integrados

📊 Rubrica de Avaliação do Projeto

Critério Peso Descrição
Planejamento 10% Topologia de rede documentada, requisitos claros
Active Directory 20% DC funcionando, OUs estruturadas, usuários criados, GPOs aplicadas
Serviços de Rede 20% DNS e DHCP funcionais e integrados
Compartilhamento 15% File Server e/ou Samba com permissões corretas
Automação 15% Scripts funcionais (Shell ou PowerShell)
Backup 10% Estratégia de backup implementada
Documentação 10% Documentação técnica completa

🎯 Requisitos Mínimos do Projeto Final

Infraestrutura Obrigatória:

  1. Windows Server:
    • Active Directory Domain Services
    • Mínimo 3 OUs organizacionais
    • Mínimo 5 usuários e 3 grupos
    • Pelo menos 2 GPOs funcionais
    • DNS integrado ao AD
    • DHCP autorizado e funcional
  2. Linux Server:
    • Servidor de arquivos Samba OU serviço web
    • Mínimo 2 scripts de automação
    • Backup automatizado via script
  3. Integração:
    • Clientes Windows autenticando no AD
    • Resolução DNS funcionando
    • DHCP distribuindo IPs
    • Compartilhamentos acessíveis

💡 DICA PARA O PROJETO: Comece pelo Active Directory! É a base de tudo. Depois configure DNS e DHCP. Só então parta para os serviços adicionais (File Server, Samba). Teste cada componente antes de passar para o próximo.
