TOPOLOGIA DA REDE — LAB VIRTUAL
Referência

Neste laboratório você vai criar 3 máquinas virtuais dentro do VirtualBox, todas rodando na mesma máquina física (o computador da Fatec ou o seu notebook). Cada VM tem dois adaptadores de rede:

📶 Adaptador 1 — NAT: conecta a VM à internet através do computador host. O VirtualBox faz o papel de roteador. Todas as VMs têm esse adaptador.
🔗 Adaptador 2 — Rede Interna (nome: Intranet): conecta as VMs entre si numa rede privada virtual 192.168.56.0/24. Esse é o "cabo de rede" entre elas dentro do VirtualBox.
🔥 Firewall UFW (192.168.56.1): atua como gateway — todo tráfego da rede interna que quiser sair para a internet passa por ele primeiro. Ele decide o que deixa passar usando as regras UFW.

🗺️ Como ler o diagrama abaixo: as setas verdes tracejadas mostram a conexão NAT (internet). As setas azuis mostram a rede interna entre as VMs através do switch virtual. A seta roxa tracejada mostra a conexão SSH do Desktop para o Server.

💡 Como configurar os adaptadores no VirtualBox: em cada VM, acesse Configurações → Rede. Adaptador 1 = NAT (acesso à internet — padrão do VirtualBox). Adaptador 2 = Rede Interna, nome: Intranet (comunicação entre as 3 VMs). O nome Intranet deve ser igual em todas as VMs — é o "switch virtual" que as conecta. O Firewall deve ter o Adaptador 2 com IP fixo 192.168.56.1 para servir como gateway.

📖 GLOSSÁRIO TÉCNICO
VirtualBox
Hypervisor tipo 2 da Oracle. Virtualiza hardware permitindo rodar múltiplos SOs num único computador.
ISO Image
Arquivo de imagem de disco que replica um CD/DVD de instalação. Usado para bootar VMs.
LVM
Logical Volume Manager — abstrai o armazenamento físico permitindo redimensionar partições em tempo de execução.
UFW
Uncomplicated Firewall — interface simplificada para gerenciar o iptables no Ubuntu.
SSH
Secure Shell — protocolo para acesso remoto seguro a servidores via linha de comando (porta 22).
NAT / Bridge
Modos de rede do VirtualBox. NAT: VM acessa internet via host. Bridge: VM recebe IP da rede física.
IP Forwarding
Parâmetro do kernel Linux que permite encaminhar pacotes entre interfaces de rede. Essencial para o firewall funcionar como gateway.
Masquerading
Tipo de NAT que substitui o IP de origem dos pacotes pelo IP da interface de saída. Permite que toda a LAN saia pela mesma interface.
netplan
Ferramenta do Ubuntu para configurar interfaces de rede via arquivos YAML em /etc/netplan/.
apt
Advanced Package Tool — gerenciador de pacotes do Ubuntu. Instala, atualiza e remove software.
PV / VG / LV
Physical Volume → Volume Group → Logical Volume. Hierarquia do LVM para organizar o armazenamento.

🖧 Instalar Ubuntu Server 22.04 — Instalação Ubuntu Server
⏱️ 40 min · Dificuldade: ⭐⭐⭐ · CLI apenas · SSH + LVM · Sem interface gráfica
+50 XP
▼
🎯 Instalar o Ubuntu Server 22.04 LTS conforme demonstrado no vídeo de referência — apenas CLI, sem GUI. Inclui LVM, usuário administrador e OpenSSH.
💡 Por que instalar sem interface gráfica?
Em segurança da informação, menos software = menor superfície de ataque. Todo serviço desnecessário é uma vulnerabilidade potencial. Servidores reais de produção rodam sem interface gráfica — você precisará dominar o terminal para administrá-los. Além disso, a instalação sem GUI consome apenas ~1 GB de RAM e deixa você no controle total do que é instalado, usando o apt para adicionar apenas o necessário.

04
🌐 Configurar Rede via Terminal — IP, Interfaces e Conectividade
⏱️ 20 min · Dificuldade: ⭐⭐ · Todas as VMs · ip · netplan · ping
+40 XP
▼
🎯 Verificar placas de rede ativas, identificar interfaces, configurar IPv4 estático via terminal e testar conectividade entre as VMs.
💡 Antes de configurar qualquer IP, o primeiro passo é sempre identificar as interfaces de rede disponíveis e verificar se estão ativas (UP) ou inativas (DOWN).
📋 LISTAR TODAS AS INTERFACES
# Mostra todas as interfaces com estado e endereço MAC $ ip link show 1: lo: <LOOPBACK,UP,LOWER_UP> mtu 65536 link/loopback 00:00:00:00:00:00 2: enp0s3: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 link/ether 08:00:27:ab:cd:ef ← Adaptador 1 (NAT) 3: enp0s8: <BROADCAST,MULTICAST> mtu 1500 link/ether 08:00:27:12:34:56 ← Adaptador 2 (Rede Interna)
⚠️ Interfaces com UP,LOWER_UP estão ativas. Sem o UP no estado = interface desligada. O nome pode variar: enp0s3, enp0s8, eth0, ens33 — use sempre o nome real que aparecer na sua máquina.
📋 VER IPs ATRIBUÍDOS ÀS INTERFACES
# Exibe IP, máscara e estado de todas as interfaces $ ip addr show 2: enp0s3: <BROADCAST,MULTICAST,UP,LOWER_UP> inet 10.0.2.15/24 brd 10.0.2.255 scope global dynamic enp0s3 3: enp0s8: <BROADCAST,MULTICAST> (sem endereço — interface inativa ou sem IP) # Versão resumida — mostra só as linhas com IP $ ip addr show | grep inet # Ver apenas uma interface específica $ ip addr show enp0s3
🔌 ATIVAR / DESATIVAR INTERFACE
# Ativar uma interface que está DOWN $ sudo ip link set enp0s8 up # Desativar uma interface $ sudo ip link set enp0s8 down # Confirmar que ficou UP $ ip link show enp0s8 3: enp0s8: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500
🗺️ VER TABELA DE ROTAS
# Exibe as rotas de rede — gateway padrão e rotas locais $ ip route show default via 10.0.2.2 dev enp0s3 proto dhcp ← Gateway padrão (internet) 10.0.2.0/24 dev enp0s3 proto kernel ← Rede local do NAT 192.168.56.0/24 dev enp0s8 proto kernel ← Rede interna das VMs
✅ Resumo dos comandos essenciais: ip link show = ver interfaces · ip addr show = ver IPs · ip route show = ver rotas · ip link set <iface> up = ativar placa
💡 No Ubuntu Server, a configuração de rede é feita via netplan — arquivos YAML em /etc/netplan/. A configuração é persistente (sobrevive ao reboot), diferente do ip addr add que é temporário.
⚡ IP TEMPORÁRIO (sem reboot — só para testes)
# Atribuir IP temporário a uma interface (some após reboot) $ sudo ip addr add 192.168.56.10/24 dev enp0s8 # Remover IP de uma interface $ sudo ip addr del 192.168.56.10/24 dev enp0s8 # Solicitar IP via DHCP manualmente $ sudo dhclient enp0s3
💾 IP PERMANENTE VIA NETPLAN (recomendado)
1

Descobrir o nome do arquivo de configuração netplan:

$ ls /etc/netplan/ 00-installer-config.yaml
2

Editar o arquivo com o nano:

$ sudo nano /etc/netplan/00-installer-config.yaml
3

Configurar IP estático na interface interna (enp0s8) — exemplo para o Ubuntu Server:

# Atenção: use ESPAÇOS, nunca TAB no YAML network: version: 2 ethernets: enp0s3: dhcp4: true # Adaptador 1 — NAT (internet via VirtualBox) enp0s8: dhcp4: false # Adaptador 2 — Rede Interna (IP fixo) addresses: - 192.168.56.20/24 # IP do Ubuntu Server na rede interna routes: - to: default via: 192.168.56.1 # Gateway = IP do Firewall nameservers: addresses: [8.8.8.8, 1.1.1.1]
4

Aplicar as configurações:

# Aplicar sem reiniciar $ sudo netplan apply # Verificar se o IP foi atribuído corretamente $ ip addr show enp0s8 3: enp0s8: <BROADCAST,MULTICAST,UP,LOWER_UP> inet 192.168.56.20/24 brd 192.168.56.255 scope global enp0s8
⚠️ Se aparecer erro ao aplicar o netplan, verifique a indentação — o YAML é sensível a espaços. Use sudo netplan try para testar antes de aplicar definitivamente.
📌 IPs padrão do laboratório:
🔥 192.168.56.1 — Firewall UFW (gateway da rede interna)
🖧 192.168.56.20 — Ubuntu Server
🖥️ 192.168.56.10 — Ubuntu Desktop (cliente)

# Testar conexão com o gateway (Firewall) $ ping -c 4 192.168.56.1 PING 192.168.56.1: 56 data bytes 64 bytes from 192.168.56.1: icmp_seq=1 ttl=64 time=0.4 ms 64 bytes from 192.168.56.1: icmp_seq=2 ttl=64 time=0.3 ms --- 192.168.56.1 ping statistics --- 4 packets transmitted, 4 received, 0% packet loss # Testar conexão com o Ubuntu Server $ ping -c 4 192.168.56.20 # Testar acesso à internet (via gateway Firewall) $ ping -c 4 8.8.8.8 # Testar resolução de nomes (DNS) $ ping -c 4 google.com PING google.com (142.250.79.46): 56 data bytes 64 bytes from 142.250.79.46: icmp_seq=1 ttl=55 time=12.1 ms

💡 IP Forwarding permite que o kernel Linux encaminhe pacotes entre interfaces. Sem ele, o tráfego da LAN não chega à internet — o firewall simplesmente descartaria os pacotes.
1

Habilitar IP forwarding permanentemente no kernel:

# Editar o arquivo de configuração do kernel fatec-fw@fatec-fw:~$ sudo nano /etc/sysctl.conf # Encontrar e descomentar (remover o #) a linha: net.ipv4.ip_forward=1 # Aplicar sem reiniciar fatec-fw@fatec-fw:~$ sudo sysctl -p net.ipv4.ip_forward = 1 # Confirmar que está ativo (deve retornar 1) fatec-fw@fatec-fw:~$ cat /proc/sys/net/ipv4/ip_forward 1
2

Configurar o masquerading (NAT) no UFW para que as VMs da LAN acessem a internet usando o IP do firewall:

# Editar as regras brutas do UFW (antes do *filter) fatec-fw@fatec-fw:~$ sudo nano /etc/ufw/before.rules # Adicionar no INÍCIO do arquivo, antes de *filter: # NAT — masquerading para saída pela interface enp0s3 (internet) *nat :POSTROUTING ACCEPT [0:0] -A POSTROUTING -s 192.168.56.0/24 -o enp0s3 -j MASQUERADE COMMIT
3

Permitir o encaminhamento (forwarding) no arquivo de configuração do UFW:

fatec-fw@fatec-fw:~$ sudo nano /etc/default/ufw # Alterar a linha DEFAULT_FORWARD_POLICY: DEFAULT_FORWARD_POLICY="ACCEPT"
⚠️ Regra de ouro: sempre libere o SSH antes de ativar o UFW. Se esquecer, perde o acesso remoto à VM sem possibilidade de recuperação pelo terminal.
## PASSO 1 — Liberar SSH e ativar UFW fatec-fw@fatec-fw:~$ sudo ufw allow 22/tcp Rules updated fatec-fw@fatec-fw:~$ sudo ufw enable Command may disrupt existing ssh connections. Proceed with operation (y|n)? y Firewall is active and enabled on system startup
## PASSO 2 — Políticas padrão (negar tudo, permitir saída) fatec-fw@fatec-fw:~$ sudo ufw default deny incoming Default incoming policy changed to 'deny' fatec-fw@fatec-fw:~$ sudo ufw default allow outgoing Default outgoing policy changed to 'allow' fatec-fw@fatec-fw:~$ sudo ufw default allow routed Default routed policy changed to 'allow' # ↑ essencial para o masquerading funcionar
## PASSO 3 — Regras para serviços permitidos fatec-fw@fatec-fw:~$ sudo ufw allow 22/tcp # SSH (já liberado, reforçando) fatec-fw@fatec-fw:~$ sudo ufw allow 80/tcp # HTTP fatec-fw@fatec-fw:~$ sudo ufw allow 443/tcp # HTTPS ## PASSO 4 — Bloquear serviços inseguros explicitamente fatec-fw@fatec-fw:~$ sudo ufw deny 23/tcp # Telnet — protocolo sem criptografia fatec-fw@fatec-fw:~$ sudo ufw deny 21/tcp # FTP — senhas em texto claro fatec-fw@fatec-fw:~$ sudo ufw deny 3389/tcp # RDP — alvo frequente de ataques
## PASSO 5 — Recarregar e verificar status completo fatec-fw@fatec-fw:~$ sudo ufw reload Firewall reloaded fatec-fw@fatec-fw:~$ sudo ufw status verbose Status: active Logging: on (low) Default: deny (incoming), allow (outgoing), allow (routed) To Action From -- ------ ---- 22/tcp ALLOW IN Anywhere 80/tcp ALLOW IN Anywhere 443/tcp ALLOW IN Anywhere 23/tcp DENY IN Anywhere 21/tcp DENY IN Anywhere 3389/tcp DENY IN Anywhere
## PASSO 6 — Testar conectividade (nas VMs Desktop/Server) # Nas outras VMs, confirmar que o gateway está correto fatec-desktop@fatec-desktop:~$ ip route show default via 192.168.56.1 dev enp0s8 ← tráfego passa pelo firewall # Testar que a internet funciona através do firewall fatec-desktop@fatec-desktop:~$ ping -c 3 8.8.8.8 3 packets transmitted, 3 received, 0% packet loss
✅ Se o ping funciona nas outras VMs, o masquerading e o IP forwarding estão funcionando. O tráfego está passando pelo firewall conforme esperado.
