TOPOLOGIA DA REDE — LAB VIRTUAL
Referência

As três VMs são criadas dentro do VirtualBox e se comunicam por meio de uma rede interna (Host-Only / Rede Interna). O Firewall UFW roda no Ubuntu Server que atua como gateway da rede interna, filtrando o tráfego entre as máquinas. Todas as VMs também possuem um Adaptador NAT para acessar a internet via o host físico.

💡 Como configurar no VirtualBox: cada VM precisa de dois adaptadores de rede. Adaptador 1 = NAT (acesso à internet). Adaptador 2 = Rede Interna ou Host-Only (comunicação entre VMs). O Firewall deve ter o Adaptador 2 configurado com IP fixo 192.168.56.1 para servir como gateway.

📖 GLOSSÁRIO TÉCNICO
VirtualBox
Hypervisor tipo 2 da Oracle. Virtualiza hardware permitindo rodar múltiplos SOs num único computador.
ISO Image
Arquivo de imagem de disco que replica um CD/DVD de instalação. Usado para bootar VMs.
LVM
Logical Volume Manager — abstrai o armazenamento físico permitindo redimensionar partições em tempo de execução.
UFW
Uncomplicated Firewall — interface simplificada para gerenciar o iptables no Ubuntu.
SSH
Secure Shell — protocolo para acesso remoto seguro a servidores via linha de comando (porta 22).
NAT / Bridge
Modos de rede do VirtualBox. NAT: VM acessa internet via host. Bridge: VM recebe IP da rede física.
IP Forwarding
Parâmetro do kernel Linux que permite encaminhar pacotes entre interfaces de rede. Essencial para o firewall funcionar como gateway.
Masquerading
Tipo de NAT que substitui o IP de origem dos pacotes pelo IP da interface de saída. Permite que toda a LAN saia pela mesma interface.
netplan
Ferramenta do Ubuntu para configurar interfaces de rede via arquivos YAML em /etc/netplan/.
apt
Advanced Package Tool — gerenciador de pacotes do Ubuntu. Instala, atualiza e remove software.
PV / VG / LV
Physical Volume → Volume Group → Logical Volume. Hierarquia do LVM para organizar o armazenamento.

💡 Por que instalar sem interface gráfica?
Em segurança da informação, menos software = menor superfície de ataque. Todo serviço desnecessário é uma vulnerabilidade potencial. Servidores reais de produção rodam sem interface gráfica — você precisará dominar o terminal para administrá-los. Além disso, a instalação sem GUI consome apenas ~1 GB de RAM e deixa você no controle total do que é instalado, usando o apt para adicionar apenas o necessário.

🔥 Configurar VM Firewall — Ubuntu Server + UFW como Gateway
⏱️ 45 min · Dificuldade: ⭐⭐⭐⭐ · 2 NICs · IP Forwarding · NAT · Regras UFW
+60 XP
▼
🎯 Criar uma VM dedicada com Ubuntu Server 22.04 mínimo, configurar duas interfaces de rede, habilitar IP forwarding e masquerading, e aplicar regras UFW para atuar como firewall/gateway da rede interna.
🔥 Firewall como VM separada — por quê?
Em arquiteturas reais, o firewall é um dispositivo dedicado que fica entre a rede interna e a internet. Neste laboratório, simulamos isso com uma VM que possui duas placas de rede: uma voltada para a internet (NAT) e outra para a rede interna das VMs. Todo o tráfego passa por ela — e ela decide o que deixa passar.

🔧 A placa de rede não funcionou após a instalação? Siga estes passos:

✅ 1. Ativar a interface manualmente

sudo ip link set enp0s3 up

✅ 2. Solicitar IP via DHCP

sudo dhclient enp0s3

🧪 3. Testar a conectividade

ping -c 4 8.8.8.8

Se responder, teste também a resolução de nomes:

ping -c 4 google.com
⚠️ Atenção: Substitua enp0s3 pelo nome real da sua interface, caso seja diferente. Use ip link show para verificar o nome correto.
⚠️ O firewall precisa de IP estático na interface interna para ser o gateway fixo das outras VMs. Use o netplan para configurar.
1

Identifique os nomes das interfaces de rede:

# Ver todas as interfaces e seus nomes admin@lab-firewall:~$ ip link show 1: lo: <LOOPBACK> ... 2: enp0s3: <BROADCAST> ... ← Adaptador 1 (NAT / internet) 3: enp0s8: <BROADCAST> ... ← Adaptador 2 (Rede Interna / LAN)
2

Edite o arquivo de configuração do netplan:

admin@lab-firewall:~$ sudo nano /etc/netplan/00-installer-config.yaml
3

Substitua o conteúdo pelo seguinte (atenção à indentação com espaços — YAML não aceita tabs):

# /etc/netplan/00-installer-config.yaml network: version: 2 ethernets: enp0s3: dhcp4: true # NAT — recebe IP do VirtualBox automaticamente enp0s8: dhcp4: false # LAN interna — IP fixo addresses: - 192.168.56.1/24 # Gateway da rede interna nameservers: addresses: [8.8.8.8, 1.1.1.1]
4

Aplique as configurações e verifique:

admin@lab-firewall:~$ sudo netplan apply admin@lab-firewall:~$ ip addr show enp0s8 3: enp0s8: <BROADCAST,MULTICAST,UP> inet 192.168.56.1/24 brd 192.168.56.255 scope global enp0s8
5

Nas VMs Desktop e Server, configure o gateway padrão como 192.168.56.1 no netplan delas (campo gateway4 ou routes).

💡 IP Forwarding permite que o kernel Linux encaminhe pacotes entre interfaces. Sem ele, o tráfego da LAN não chega à internet — o firewall simplesmente descartaria os pacotes.
1

Habilitar IP forwarding permanentemente no kernel:

# Editar o arquivo de configuração do kernel admin@lab-firewall:~$ sudo nano /etc/sysctl.conf # Encontrar e descomentar (remover o #) a linha: net.ipv4.ip_forward=1 # Aplicar sem reiniciar admin@lab-firewall:~$ sudo sysctl -p net.ipv4.ip_forward = 1 # Confirmar que está ativo (deve retornar 1) admin@lab-firewall:~$ cat /proc/sys/net/ipv4/ip_forward 1
2

Configurar o masquerading (NAT) no UFW para que as VMs da LAN acessem a internet usando o IP do firewall:

# Editar as regras brutas do UFW (antes do *filter) admin@lab-firewall:~$ sudo nano /etc/ufw/before.rules # Adicionar no INÍCIO do arquivo, antes de *filter: # NAT — masquerading para saída pela interface enp0s3 (internet) *nat :POSTROUTING ACCEPT [0:0] -A POSTROUTING -s 192.168.56.0/24 -o enp0s3 -j MASQUERADE COMMIT
3

Permitir o encaminhamento (forwarding) no arquivo de configuração do UFW:

admin@lab-firewall:~$ sudo nano /etc/default/ufw # Alterar a linha DEFAULT_FORWARD_POLICY: DEFAULT_FORWARD_POLICY="ACCEPT"
⚠️ Regra de ouro: sempre libere o SSH antes de ativar o UFW. Se esquecer, perde o acesso remoto à VM sem possibilidade de recuperação pelo terminal.
## PASSO 1 — Liberar SSH e ativar UFW admin@lab-firewall:~$ sudo ufw allow 22/tcp Rules updated admin@lab-firewall:~$ sudo ufw enable Command may disrupt existing ssh connections. Proceed with operation (y|n)? y Firewall is active and enabled on system startup
## PASSO 2 — Políticas padrão (negar tudo, permitir saída) admin@lab-firewall:~$ sudo ufw default deny incoming Default incoming policy changed to 'deny' admin@lab-firewall:~$ sudo ufw default allow outgoing Default outgoing policy changed to 'allow' admin@lab-firewall:~$ sudo ufw default allow routed Default routed policy changed to 'allow' # ↑ essencial para o masquerading funcionar
## PASSO 3 — Regras para serviços permitidos admin@lab-firewall:~$ sudo ufw allow 22/tcp # SSH (já liberado, reforçando) admin@lab-firewall:~$ sudo ufw allow 80/tcp # HTTP admin@lab-firewall:~$ sudo ufw allow 443/tcp # HTTPS ## PASSO 4 — Bloquear serviços inseguros explicitamente admin@lab-firewall:~$ sudo ufw deny 23/tcp # Telnet — protocolo sem criptografia admin@lab-firewall:~$ sudo ufw deny 21/tcp # FTP — senhas em texto claro admin@lab-firewall:~$ sudo ufw deny 3389/tcp # RDP — alvo frequente de ataques
## PASSO 5 — Recarregar e verificar status completo admin@lab-firewall:~$ sudo ufw reload Firewall reloaded admin@lab-firewall:~$ sudo ufw status verbose Status: active Logging: on (low) Default: deny (incoming), allow (outgoing), allow (routed) To Action From -- ------ ---- 22/tcp ALLOW IN Anywhere 80/tcp ALLOW IN Anywhere 443/tcp ALLOW IN Anywhere 23/tcp DENY IN Anywhere 21/tcp DENY IN Anywhere 3389/tcp DENY IN Anywhere
## PASSO 6 — Testar conectividade (nas VMs Desktop/Server) # Nas outras VMs, confirmar que o gateway está correto admin@lab-desktop:~$ ip route show default via 192.168.56.1 dev enp0s8 ← tráfego passa pelo firewall # Testar que a internet funciona através do firewall admin@lab-desktop:~$ ping -c 3 8.8.8.8 3 packets transmitted, 3 received, 0% packet loss
✅ Se o ping funciona nas outras VMs, o masquerading e o IP forwarding estão funcionando. O tráfego está passando pelo firewall conforme esperado.
