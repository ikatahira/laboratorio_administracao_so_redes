🛡️ POR QUE ESTA AULA É ESSENCIAL PARA O TECNÓLOGO EM SEGURANÇA DA INFORMAÇÃO?

🌐
Controle de Rede

Saber configurar interfaces de rede é pré-requisito para qualquer atuação em infraestrutura segura. Sem rede, não há comunicação; sem controle, não há segurança.

🔥
Firewall é a 1ª Linha de Defesa

O UFW implementa o conceito de Defesa em Profundidade — bloquear portas desnecessárias reduz a superfície de ataque. É o que separa um servidor exposto de um servidor protegido.

🏢
Mercado de Trabalho

Analistas de SI, pentesters e administradores de redes usam esses comandos diariamente. Dominar Linux e firewalls é requisito básico para cargos como Analista de Segurança e SOC Analyst.

🔒
Princípio do Menor Privilégio

Bloquear tudo e liberar apenas o necessário (deny incoming + allow 22/tcp) é o princípio fundamental da segurança: nunca exponha o que não precisa.

🕵️
Análise de Vulnerabilidades

Entender portas e protocolos é base para ferramentas como Nmap, Nessus e Wireshark. Saber o que deve estar aberto (e fechado) é o primeiro passo de qualquer pentest.

📜
Normas e Compliance

A norma ISO 27001 e o LGPD exigem controles de acesso a sistemas. Configurar firewall e usuários corretamente é parte da conformidade com essas normas.

🌐 Configurando a Rede com Netplan
💡 O que é o Netplan?
O Netplan é a ferramenta do Ubuntu para configurar interfaces de rede. O arquivo de configuração fica em /etc/netplan/ e usa o formato YAML — que exige recuo (indentação) com espaços exatos. Um erro de espaço quebra tudo!

Figura: Placa NAT usa DHCP · Placa Interna usa IP fixo classe C

Esta configuração é para a placa em modo NAT — o VirtualBox atribui o IP automaticamente.
1
Acessar o arquivo de configuração da rede
Caminho: /etc/netplan/

Primeiro, veja qual arquivo existe dentro da pasta netplan:

$ ls /etc/netplan/
📺 Resultado esperado:
00-installer-config.yaml

Agora abra o arquivo para editar com o nano:

$ sudo nano /etc/netplan/00-installer-config.yaml # O nano abrirá o arquivo. Você verá o conteúdo atual na tela.
⚠️ O comando começa com sudo porque editar arquivos em /etc/ requer permissão de administrador. O sistema pode pedir sua senha.
2
Digitar a configuração DHCP
Atenção: use ESPAÇOS, nunca Tab!

Apague o conteúdo atual (se houver) e cole/digite exatamente assim:

# /etc/netplan/00-installer-config.yaml network: version: 2 ethernets: enp0s3: dhcp4: true
📐 Mapa de espaços — conte exatamente assim:

network:                          ← nível 0 — sem espaço antes
··version: 2                   ← nível 1 — 2 espaços antes
··ethernets:                  ← nível 1 — 2 espaços antes
····enp0s3:                    ← nível 2 — 4 espaços antes
······dhcp4: true              ← nível 3 — 6 espaços antes
⚠️ Use espaços, nunca a tecla Tab. Um espaço errado causa erro ao aplicar.
3
Salvar o arquivo no nano
Sequência de teclas: Ctrl+X → Y → Enter

No nano, use esta sequência para salvar e sair:

  • Pressione Ctrl + X → o nano pergunta se deseja salvar
  • Pressione Y (Yes) → confirmar que quer salvar
  • Pressione Enter → confirmar o nome do arquivo

Você voltará para o terminal normal.

4
Aplicar a configuração
Sem reiniciar a máquina!
$ sudo netplan apply # Aguarde alguns segundos. Se não aparecer erro, está correto.
⚠️ Se aparecer um erro como "invalid YAML", provavelmente há um problema de indentação. Abra o arquivo novamente com nano e verifique os espaços.
5
Verificar se o IP foi recebido
Confirmar que o DHCP funcionou
$ ip addr show enp0s3
📺 Resultado esperado (sucesso):
2: enp0s3: <BROADCAST,MULTICAST,UP,LOWER_UP> link/ether 08:00:27:xx:xx:xx brd ff:ff:ff:ff:ff:ff inet 10.0.2.15/24 brd 10.0.2.255 scope global dynamic enp0s3 ← ✅ IP recebido! valid_lft 86391sec preferred_lft 86391sec
✅ Você verá uma linha inet 10.0.2.15/24 (ou similar começando com 10.0.2.) — isso significa que o DHCP funcionou e sua placa NAT tem acesso à internet!

Teste a conexão com a internet:

$ ping -c 4 8.8.8.8 # Envia 4 pacotes para o DNS do Google. Se responder, há internet!
📺 Resultado esperado:
PING 8.8.8.8 (8.8.8.8) 56(84) bytes of data. 64 bytes from 8.8.8.8: icmp_seq=1 ttl=118 time=12.4 ms 64 bytes from 8.8.8.8: icmp_seq=2 ttl=118 time=11.8 ms --- 8.8.8.8 ping statistics --- 4 packets transmitted, 4 received, 0% packet loss ← ✅ perfeito!
Esta configuração é para a placa Rede Interna. O IP fixo permite que outras VMs usem esse servidor como gateway.
1
Descobrir o nome da interface de rede interna
Pode ser enp0s8 ou outro nome
$ ip link show # Procure pelas interfaces que NÃO são lo (loopback)
📺 Resultado esperado:
1: lo: <LOOPBACK> ... ← ignorar (loopback) 2: enp0s3: <BROADCAST...> ... ← placa NAT 3: enp0s8: <BROADCAST...> ... ← ✅ placa Rede Interna
💡 O nome pode ser diferente no seu computador (ex: eth1, ens8). Anote o nome correto e use-o no próximo passo.
2
Editar o arquivo netplan com IP fixo
IP: 192.168.10.1 — Classe C privada
$ sudo nano /etc/netplan/00-installer-config.yaml

Digite a configuração abaixo (substitua enp0s8 pelo nome da sua interface se for diferente):

# /etc/netplan/00-installer-config.yaml network: version: 2 ethernets: enp0s8: dhcp4: false addresses: - 192.168.10.1/24
📐 Mapa de espaços — conte exatamente assim:

network:                          ← nível 0 — sem espaço antes
··version: 2                   ← nível 1 — 2 espaços antes
··ethernets:                  ← nível 1 — 2 espaços antes
····enp0s8:                    ← nível 2 — 4 espaços antes
······dhcp4: false             ← nível 3 — 6 espaços antes
······addresses:                ← nível 3 — 6 espaços antes
········- 192.168.10.1/24       ← nível 4 — 8 espaços + traço
⚠️ Use espaços, nunca a tecla Tab. Um espaço errado causa erro ao aplicar.
📐 Por que 192.168.10.1/24? O IP 192.168.10.1 é o gateway da rede interna. O /24 significa máscara 255.255.255.0 — ou seja, todos os IPs de 192.168.10.1 a 192.168.10.254 estão na mesma rede.
3
Salvar e aplicar
# Salvar no nano: Ctrl+X → Y → Enter # Depois aplicar: $ sudo netplan apply
4
Verificar se o IP fixo foi atribuído
$ ip addr show enp0s8
📺 Resultado esperado (sucesso):
3: enp0s8: <BROADCAST,MULTICAST,UP,LOWER_UP> link/ether 08:00:27:yy:yy:yy brd ff:ff:ff:ff:ff:ff inet 192.168.10.1/24 scope global enp0s8 ← ✅ IP fixo configurado!
✅ As outras VMs da Rede Interna devem apontar para 192.168.10.1 como gateway e DNS — esse é o endereço do nosso servidor!
Configuração completa: enp0s3 via DHCP (internet) + enp0s8 com IP fixo (rede interna).

⚡ Habilitando o IP Forwarding
💡 O que é IP Forwarding?
Permite que o Ubuntu "repasse" pacotes entre interfaces. Sem isso, a VM não funciona como roteador/gateway — os pacotes chegam pela Rede Interna mas não saem pela NAT. É como habilitar a "função de roteador" no servidor.
1
Abrir o arquivo de configuração do kernel
Arquivo: /etc/sysctl.conf
$ sudo nano /etc/sysctl.conf # O arquivo é longo. Use Ctrl+W para buscar a linha que precisa
2
Buscar e descomentar a linha correta
Use Ctrl+W no nano para buscar

No nano, pressione Ctrl + W, digite ip_forward e pressione Enter. Você encontrará esta linha:

# Antes (linha comentada — o # bloqueia): #net.ipv4.ip_forward=1 # Depois (remova o # do início): net.ipv4.ip_forward=1
⚠️ Remova apenas o # da linha net.ipv4.ip_forward=1. Não altere mais nada no arquivo. Se a linha não existir, adicione-a no final do arquivo.
3
Salvar e aplicar sem reiniciar
# Salvar no nano: Ctrl+X → Y → Enter # Aplicar as mudanças imediatamente (sem reiniciar): $ sudo sysctl -p
📺 Resultado esperado:
net.ipv4.ip_forward = 1 ← ✅ confirmado que está ativo!
4
Confirmar que está ativo
$ cat /proc/sys/net/ipv4/ip_forward
📺 Resultado esperado:
1 ← ✅ 1 = ativo, 0 = inativo
✅ O valor 1 confirma que o IP Forwarding está ativo. O servidor agora consegue rotear pacotes entre a internet (NAT) e a rede interna.

🔥 Configurando o Firewall UFW
💡 O que é o UFW?
UFW = Uncomplicated Firewall. É a ferramenta do Ubuntu para controlar quais conexões entram e saem do servidor. Ele cria uma "parede de fogo" virtual que filtra o tráfego de rede por portas e protocolos.

Figura: O UFW filtra o tráfego antes de chegar ao servidor

🔐 Passo a passo — Configurar o UFW do zero
🚨 CUIDADO CRÍTICO: Se você acessa o servidor via SSH, LIBERE A PORTA 22 ANTES de ativar o UFW. Se você ativar o UFW sem liberar a porta 22, você perde o acesso remoto e precisará acessar fisicamente a máquina!
1
OBRIGATÓRIO: Liberar SSH (porta 22) antes de tudo
Sem isso, você perde o acesso à máquina!
# ⚠️ FAÇA ISSO PRIMEIRO — liberar acesso SSH $ sudo ufw allow 22/tcp
📺 Resultado esperado:
Rules updated Rules updated (v6) ← ✅ regra criada para IPv4 e IPv6
2
Definir a política padrão
Bloquear tudo que entra, permitir tudo que sai
# Bloquear tudo que vem de fora (incoming) $ sudo ufw default deny incoming # Permitir que a máquina se conecte a outros serviços $ sudo ufw default allow outgoing
🛡️ Conceito de Segurança: "Bloquear tudo e liberar apenas o necessário" é o Princípio do Menor Privilégio — fundamental na Segurança da Informação. Começamos negando tudo e abrimos apenas o que precisamos.
3
Ativar o UFW
$ sudo ufw enable
📺 O sistema vai perguntar:
Command may disrupt existing ssh connections. Proceed with operation (y|n)? y ← digite y e Enter Firewall is active and enabled on system startup ← ✅ UFW ativado!
✅ O UFW agora está ativo e iniciará automaticamente toda vez que o servidor ligar.
4
Liberar portas necessárias
# Liberar HTTP (sites sem criptografia) $ sudo ufw allow 80/tcp # Liberar HTTPS (sites com criptografia — o correto!) $ sudo ufw allow 443/tcp # Bloquear Telnet (porta 23 — transmite tudo sem criptografia!) $ sudo ufw deny 23/tcp # Bloquear FTP (porta 21 — protocolo de transferência inseguro) $ sudo ufw deny 21/tcp # Bloquear RDP (porta 3389 — área de trabalho remota Windows) $ sudo ufw deny 3389/tcp
💡 Por que bloquear Telnet (porta 23)? O Telnet transmite usuário, senha e todos os dados em texto puro — qualquer pessoa na rede pode interceptar com o Wireshark. O SSH (porta 22) é o substituto seguro com criptografia.
5
Ver o status das regras
Verificar tudo que está configurado
# Ver status básico $ sudo ufw status # Ver com mais detalhes (recomendado) $ sudo ufw status verbose # Ver regras numeradas (útil para remover depois) $ sudo ufw status numbered
📺 Resultado esperado após configuração completa:
Status: active To Action From -- ------ ---- [ 1] 22/tcp ALLOW IN Anywhere ← SSH liberado [ 2] 80/tcp ALLOW IN Anywhere ← HTTP liberado [ 3] 443/tcp ALLOW IN Anywhere ← HTTPS liberado [ 4] 23/tcp DENY IN Anywhere ← Telnet bloqueado [ 5] 21/tcp DENY IN Anywhere ← FTP bloqueado [ 6] 3389/tcp DENY IN Anywhere ← RDP bloqueado
6
Como remover uma regra se errar
# Passo 1: ver regras com número $ sudo ufw status numbered # Passo 2: deletar a regra pelo número (exemplo: deletar regra 4) $ sudo ufw delete 4 # Passo 3: confirmar com y # Se precisar desativar o UFW completamente (emergência): $ sudo ufw disable # Para reativar depois: $ sudo ufw enable
